# GAO Agile Assessment Guide Glossary

Key terms from the GAO Agile Assessment Guide (GAO-24-105506), alphabetical, with
the chapter(s) where each is developed. Definitions are synthesized, not quoted.

- **Acceptance criteria** — the product owner's specific conditions for a single user
  story; what that one story must satisfy to be accepted. (ch05)
- **Affinity estimation** — a consensus technique for sizing relative effort and
  exposing the factors that drive a story's complexity. (ch03)
- **Agile Manifesto** — the 2001 statement (17 authors) of four value pairs; with its
  12 principles it is the invariant an organization's Agile use is judged against. (ch01)
- **Architectural runway** — keeping the technical infrastructure, interfaces, and
  dependencies understood and in place just ahead of near-term work, extended
  continually so large up-front design is avoided. (ch03)
- **Authority to Operate (ATO)** — a risk-based security authorization gate; Agile teams
  streamline it by partnering with security early and building docs iteratively. (ch06)
- **Backlog (prioritized)** — the value-ordered, ever-living list of user stories, bugs,
  and defects; never empty while the program exists, and a key traceability artifact. (ch05)
- **Burn up / burn down chart** — progress charts; burn up plots cumulative points
  completed over time, burn down plots remaining work over time. (ch08)
- **Cadence** — the steady rhythm of fixed-length iterations (commonly 2–4 weeks) and
  fixed-interval releases that makes an Agile program predictable. (ch04)
- **Continuous integration (CI)** — frequently merging developers' working copies via an
  automated build that runs functional and non-functional tests. (ch03, ch05)
- **Control account** — the smallest building block of the performance measurement
  baseline in EVM; each is built on a feature (or group of features). (ch07)
- **Critical (mission/architectural/safety) features** — features identified early because
  defining them late risks heavy rework and integration trouble. (ch03, ch09)
- **Cumulative flow diagram (CFD)** — banded chart, one band per workflow stage, that
  visualizes flow and exposes bottlenecks. (ch08)
- **Cycle time** — elapsed time from starting work on a story to getting it into
  production; teams want it short. (ch08)
- **Definition of done** — the universal gate applied to every user story: acceptance plus
  bundled activities such as testing and compliance; multiple teams should share one. (ch05)
- **Definition of ready** — the detail a story needs before work may start (e.g., it is
  estimated and its acceptance criteria are written). (ch05)
- **Deployment** — making functionality available to external users; distinct from a
  release, which is often an internal hand-off of working code. (ch04)
- **DevOps / DevSecOps** — collaboration across dev, operations, and QA for frequent small
  releases; DevSecOps folds security into every stage as a shared responsibility. (ch01, ch09)
- **Earned value management (EVM)** — measuring planned vs. earned vs. actual; in Agile it
  is taken at the feature level, tailored to program size, and read on the Agile cadence. (ch07)
- **Epic** — a high-level, low-detail capability spanning multiple iterations before it is
  split into stories; a level stable enough to baseline. (ch04, ch05, ch07)
- **Estimate at complete (EAC)** — projected total at finish; in Agile expressed in effort
  (completed plus remaining work divided by an efficiency index). (ch07)
- **Feature** — a chunk of work completable in one or two reporting periods, detailed
  enough to stay stable; the level tracked in the cost estimate, schedule, and EVM. (ch05, ch07)
- **Freeze period** — a window during which in-progress baseline work is shielded from
  change so measurement stays credible. (ch07)
- **Functional vs. non-functional requirements** — user-facing capability vs. qualities
  like security, privacy, reliability, scalability, and Section 508 accessibility. (ch03, ch05)
- **Information radiator** — a visible, accessible display (whiteboard or screen) that
  circulates performance information to everyone on a program. (ch08)
- **INVEST** — qualities of a good user story: Independent, Negotiable, Valuable,
  Estimable, Small, Testable. (ch03, ch09)
- **Integrated master schedule (IMS)** — the single schedule capturing all features
  (Agile and non-Agile, government and contractor) using rolling wave planning. (ch07)
- **Iteration** — a time-boxed, recurring window that produces working software; a release
  is a planning segment of usable requirements. (ch01, ch04)
- **Kanban** — continuous-flow method using a board and work-in-progress limits to surface
  bottlenecks; deemphasizes time-boxing. (ch01, ch05, ch09)
- **Lead time** — elapsed time from identifying a capability to releasing it into
  production. (ch08)
- **Minimum viable product (MVP)** — a product built with least effort but still rich
  enough for real customer interaction and learning; not merely the smallest slice. (ch04, ch05)
- **Modular contracting** — the FAR-authorized practice (41 U.S.C. § 2308 / FAR 39.103) of
  meeting a need through successive interoperable increments; the contracting fit for
  incremental delivery. (ch06)
- **MoSCoW** — a prioritization scheme: Must have, Should have, Could have, Won't have this
  time. (ch03, ch09)
- **Performance measurement baseline (PMB)** — the time-phased budget against which EVM
  measures; its stability (not iteration flexibility) is what protects EVM's value. (ch07)
- **Product owner** — the empowered government representative who owns and orders the
  backlog, sets acceptance criteria, and decides "done"; over-stretching this role across
  too many teams is a named failure mode. (ch03, ch05, ch06)
- **Quantifiable backup data (QBD)** — feature-progress information judged against a
  feature's acceptance criteria; the bridge from volatile story data to feature-level EVM. (ch07)
- **Relative estimation** — sizing stories by comparison (e.g., point scales, t-shirt
  sizes) rather than in absolute time; team-specific and not portable across teams. (ch03, ch07)
- **Release** — typically an internal hand-off of working code; in government it may go to a
  certifier or independent tester, not straight to the user. (ch04)
- **Retrospective** — an end-of-iteration reflection on what to improve; findings are owned
  by the team and can become backlog stories, and they feed contract oversight. (ch03, ch06)
- **Rolling wave planning** — planning near-term work in detail and later work coarsely;
  Agile's progressive-definition style, bounded by the road map. (ch04, ch07)
- **Self-organizing team** — a team empowered to own the whole product and decide how work
  gets done; failing to self-organize risks inefficiency and slips. (ch03)
- **Spike** — an XP-derived placeholder story for research a team must do before it can size
  an unfamiliar story. (ch05)
- **Story mapping** — arranging user activities by priority and sophistication so slices
  ship usable, not high-value features stranded on deferred dependencies. (ch03)
- **Statement of Objectives (SOO) / Performance Work Statement (PWS) / Statement of Work
  (SOW)** — solicitation documents on a certainty spectrum; prefer a SOO or PWS when the
  end state will evolve, a SOW only when it is well understood. (ch06)
- **Sustainable pace** — a cadence a team can hold indefinitely; the source of credible
  velocity data and a guard against burnout. (ch03)
- **Technical debt** — accumulated code deficiencies and deferred detail whose "interest"
  rises until teams fix errors instead of building features. (ch03, ch09)
- **Time-boxing** — fixing an iteration's length; not the same as a hard schedule date
  constraint, which can break traceability and the critical path. (ch04, ch07)
- **Traceability** — linking source requirements to lower-level items and back; in Agile the
  road map and backlog replace the Waterfall requirements traceability matrix. (ch05, ch08)
- **User story** — the smallest, most volatile unit of work, told from the user's view
  (who needs it, and why); weighted in story points. (ch03, ch05, ch07)
- **Value Stream Mapping (VSM)** — a flowchart of the steps, people, and information used to
  deliver value, used to cut hand-offs and cycle time. (ch07)
- **Velocity** — work a team finishes per iteration (in points); a team-internal planning
  aid and estimate-refinement input — never a cross-team comparison or a target. (ch03, ch07, ch08)
- **Vision → epic → release → iteration → user story** — the five Agile planning levels,
  an inverted triangle that keeps detail traceable from strategy down to coded work. (ch04)
- **Work breakdown structure (WBS)** — the single product-oriented decomposition tying
  epics → features → user stories; the shared spine feeding cost, schedule, and EVM. (ch07)
- **Waterfall** — the sequential model (requirements → design → build → test) used as
  Agile's contrast case. (ch01, ch04)
