# Chapter 5: Implement and Assess Steps

> Scope caveat: This chapter covers two steps of the NIST Risk Management Framework (RMF) — Implement (3.4) and Assess (3.5). The RMF is a security-and-privacy governance overlay that *consumes* systems-engineering outputs and runs in parallel with the SDLC. Treat it as the security/RMF edge of systems engineering: it tells you *when and how to prove controls work and document deficiencies*, not how to engineer the system itself. Where it touches core SE, it does so by pointing at SP 800-160 v1 processes (implementation, integration, verification, validation, configuration management).

## Core Idea
Once controls have been selected and planned, two RMF steps turn the plan into demonstrable reality. **Implement** puts the controls from the security and privacy plans into the system and captures, in a baseline configuration, exactly how each was built. **Assess** then determines whether those controls were built correctly, are working as designed, and actually deliver the intended result against the system's security and privacy requirements. The pairing is deliberate: you build to the plan, record the "as-implemented" delta, then have someone with appropriate independence test the result and document every finding. The chapter's through-line is that engineering best practices, early/iterative testing, evidence reuse, and honest documentation of deficiencies are what make this efficient rather than a late-stage compliance gate.

## Frameworks Introduced
The slice is structured as a fixed task list per step. These are the named constructs, with when/how they apply:

- **Implement step, Task I-1 (Control Implementation)** — implement the controls described in the security and privacy plans, consistent with the enterprise, security, and privacy architectures. Primary responsibility: System Owner and Common Control Provider. Apply during New (Development/Acquisition; Implementation/Assessment) and Existing (Operations/Maintenance) lifecycle phases.
- **Implement step, Task I-2 (Update Control Implementation Information)** — record changes between the planned and the "as-implemented" state of controls, producing updated plans and a system configuration baseline. Applies because plans rarely survive contact with implementation unchanged.
- **Assess step, Task A-1 (Assessor Selection)** — the Authorizing Official (or designated representative) selects the assessor or team, weighing technical expertise and required independence.
- **Assess step, Task A-2 (Assessment Plan)** — assessors develop assessment plans (objectives and per-control procedures); the AO reviews and approves them.
- **Assess step, Task A-3 (Control Assessments)** — the Control Assessor runs the procedures against the implemented controls and produces assessment evidence.
- **Assess step, Task A-4 (Assessment Reports)** — the Control Assessor documents findings and recommendations; the report is a key artifact in the authorization package.
- **Assess step, Task A-5 (Remediation Actions)** — fix what can be fixed quickly, then reassess; update reports and plans.
- **Assess step, Task A-6 (Plan of Action and Milestones)** — capture remaining deficiencies as a prioritized, scheduled remediation plan in the authorization package.
- **Cybersecurity Framework cross-references** — the slice tags Implement outcomes to CSF constructs (e.g., PR.IP-1, PR.IP-2, Profile) and Assess outcomes to Profile and ID.RA-6. Use these when mapping RMF activity onto an organization's CSF Profile.
- **SP 800-160 v1 process call-outs** — each task's References point at specific SP 800-160 v1 processes: Implementation, Integration, Verification, Transition for I-1; those plus Configuration Management for I-2; Verification and Validation for the Assess tasks. This is the explicit bridge from RMF governance to systems-security-engineering practice.

## Key Concepts
- **The assessment criterion (the "three-part test").** Across the Assess step, a control passes only on three conditions: it was built correctly, it operates as designed, and it yields the result needed to satisfy the system's and organization's security and privacy requirements. This same triple defines the purpose of assessment, the goal of a control assessment (A-3), and the goal of a reassessment after remediation (A-5).
- **Plan vs. as-implemented.** It is not always feasible to build controls exactly as planned. Task I-2 exists to capture revised descriptions — changed inputs, expected behavior, and outputs — in enough detail for assessors to work from. Recording the as-implemented state is what later lets you detect control changes, judge whether they were authorized, and gauge their effect on the security and privacy posture.
- **The configuration baseline.** Implementation produces a documented baseline configuration of the control implementation. This is both an Implement-step output and the anchor for later change detection.
- **Common vs. system-specific vs. hybrid controls.** Controls the system *inherits* (common controls) are coordinated with the common control provider and assessed separately by that provider's assessors — they need not be re-assessed at the system level. When inherited controls turn out not to meet the system's requirements, the system owner adds compensating or supplementary controls (system-specific or hybrid), or accepts the greater risk with organizational approval.
- **Assurance requirements.** Beyond functional implementation, organizations address assurance — the activities developers and implementers perform to raise confidence that controls are correct, working, and effective. Assurance is about the *quality* of design, development, and implementation, and SP 800-53 lists assurance-related controls.
- **COTS and third-party validation.** Where the organization has no direct control over a system element's internal controls (e.g., commercial off-the-shelf products), it favors elements tested or validated by approved independent labs (the slice names NIST Cryptographic Module Validation Program labs and NIAP Common Criteria labs). Crucially, those validations cover the product *in isolation and in a specific configuration*; implementation must address how the product is integrated while preserving security functionality and assurance.
- **Assessor independence.** Self-assessment is allowed, but the AO sets the required independence level based on law, policy, and standards, consulting the Inspector General, CIO, SAOP, and SAISO. Independence means freedom from real or perceived conflicts of interest regarding control effectiveness or the system's development/operation/management. Independence during continuous monitoring is what enables reuse of results for ongoing authorization.
- **Privacy as a parallel track.** The Senior Agency Official for Privacy is accountable for the privacy program: the SAOP serves as the control assessor for privacy controls, conducts the initial privacy-control assessment before operation and periodic reassessments, and provides privacy information to the AO. Independent privacy assessment is discretionary, not mandated (OMB A-130). Where controls serve both security and privacy objectives, security and privacy assessors collaborate.
- **Findings, deficiencies, and vulnerabilities are distinct.** Findings are a factual report of whether controls operate as intended. A deficiency becomes a *vulnerability* only when it can be exploited by a threat agent — the slice draws this line explicitly.
- **Interim reports and the evolving assessment record.** Assessment results from earlier SDLC phases go into interim reports that feed the final report, reinforcing that assessment reports are living documents. An optional executive summary gives the AO an abbreviated synopsis.
- **The addendum (right of reply).** After initial findings, system owners/common control providers may attach an addendum to respond — adding context, rebutting findings, or correcting the record. The addendum is considered by the AO but does *not* change the original assessor findings.
- **Reassessment preserves the original record.** When remediated controls are reassessed, assessors *add* the reassessment findings to the report but do not overwrite the original results. By the end of assessment, the plans accurately describe every control as built, compensating controls included.
- **The POA&M.** The Plan of Action and Milestones lists planned corrections for deficiencies (from assessment and from continuous monitoring): tasks, a recommendation to complete before or after authorization, required resources, milestones, and scheduled completion dates. Risk assessment and security categorization drive its prioritization. Deficiencies the AO accepts as residual risk need no POA&M entry, but are still documented in the assessment reports for the audit trail.

## Mental Models
- **Build to the plan, then write down what really happened.** Implement is two motions: install the controls (I-1), then reconcile plan-vs-reality into the baseline (I-2). The gap between "designed" and "as-built" is not failure — it is the thing you must capture, because everything downstream (change detection, assessment, authorization) reads the as-implemented record.
- **Assessment is the verification/validation lane of RMF.** The slice keeps pointing the Assess tasks at SP 800-160 v1 Verification and Validation. Think of A-1→A-6 as RMF's V&V wrapper: pick the assessor, plan the test, run it, report it, fix the quick wins, and schedule the rest.
- **Shift left to save money.** Conduct initial control assessments *during* development, in parallel with the SDLC. Developmental testing and evaluation — design and code reviews, regression testing, application scanning — catches deficiencies when they are cheapest to fix, and the results can be reused at authorization to avoid costly repetition.
- **Iterative system, iterative assessment.** For agile/iterative development, assess controls per cycle as each completes; the same incremental pattern applies to COTS components and lets you start assessing before every planned control is fully in place when that is more efficient.
- **Reuse is the efficiency engine.** Evidence can come from prior audits, vendor/developer assessments, external product-evaluation programs (Common Criteria, CMVP), earlier SDLC testing (unit/functional/acceptance), and reciprocity (authorization-to-use). Establish reuse criteria up front in policy or program plans so reuse is principled, not ad hoc.
- **Automate the assessment, not just the controls.** The slice repeatedly pushes maximizing automation in control assessments to increase speed, effectiveness, and efficiency, and to feed near-real-time / continuous monitoring.
- **Two destinies for a deficiency.** Fix it now (initial remediation + reassess, A-5) or schedule it (POA&M, A-6) — and a third path: the AO accepts it as residual risk, which still gets logged but generates no milestone.
- **Severity routing by risk and mission.** Prioritization is not first-come-first-served: risk assessment, security categorization, and mission criticality decide what gets remediated first, with senior leadership ensuring resources flow to the most critical missions and highest-risk deficiencies.

## Anti-patterns
The source does not name explicit anti-patterns. The closest source-grounded cautions, stated as practices to avoid the opposite of:
- **Treating product validation as system validation.** Relying on an isolated lab evaluation of a COTS element while ignoring how integration into the system affects its security functionality and assurance — the slice explicitly warns the validation covered the product in isolation and in a specific configuration only.
- **Assessing late.** Deferring all assessment to a single end-of-SDLC gate forfeits the cheaper, earlier corrections and the reuse value the slice describes.

## Key Takeaways
1. **Implement = build + record.** Put the planned controls in (I-1) and capture the as-implemented delta in updated plans and a configuration baseline (I-2).
2. **Assess against one criterion:** implemented correctly, operating as intended, producing the desired outcome — for the system's and organization's security and privacy requirements.
3. **Six Assess tasks form a V&V loop:** select assessor (A-1), plan (A-2), assess (A-3), report (A-4), remediate-and-reassess (A-5), POA&M the rest (A-6).
4. **Independence is set by the AO**, scaled to law and policy; self-assessment is permitted; independence enables result reuse for ongoing authorization.
5. **Shift left and reuse:** assess during development, prefer iterative/incremental assessment, automate, and reuse prior evidence under pre-established criteria.
6. **Deficiencies are routed by risk:** fix-now, schedule via POA&M, or AO-accepted residual risk — all documented for the audit trail; only exploitable deficiencies are vulnerabilities.
7. **Privacy runs in parallel**, with the SAOP accountable for assessing privacy controls and informing the AO.
