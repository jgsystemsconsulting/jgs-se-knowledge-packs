# Chapter 5: Human Reliability Analysis and Software Risk Assessment

## Core Idea
Humans and software are two of the most consequential — and most poorly quantified — contributors to space-system risk, so the PRA Guide gives each its own quantitative treatment. Human Reliability Analysis (HRA) turns crew, ground-crew, and maintenance interactions into human basic events with assigned human error probabilities (HEPs) that drop directly into the event-tree and fault-tree models. Software Risk Assessment, via the Context-based Software Risk Model (CSRM), reframes software failure away from a single time-driven failure rate and toward *conditional* failures: the software did exactly what it was designed to do, but that response was wrong for the mission condition it encountered. Both methods exist to make the PRA realistic rather than optimistic, and both feed the same Boolean logic models so that human, software, and hardware contributions are assessed at comparable depth. This is where `nasa-risk`'s qualitative risk vocabulary becomes a number you can defend.

## Frameworks Introduced

- **HRA five-step process** (Section 8.1): The backbone of any HRA — Problem Definition, Task Analysis, Error Identification, Error Representation (Modeling), and Quantification and Integration into PRA. Use it as the standing procedure whenever human-system interactions (HSIs) are numerous enough to materially affect the risk estimate. Error Identification is named the most important step.
- **Pre-initiator / Initiator / Post-initiator HSI classification** (Section 8.2.1, after Swain and EPRI NP-6560L): Categorises human interactions by their timing relative to the accident initiating event. Use it as the first cut when scoping which human actions to model and where in the logic model they belong.
- **Skill / Rule / Knowledge-based response** (Section 8.2.2, Rasmussen): Three categories of cognitive response, ordered by increasing error-proneness. Use it when characterising the cognitive demand of a diagnosis or decision task.
- **Error of Omission / Error of Commission** (Section 8.2.3, Swain): EOM is failing to perform a required action; ECOM is performing it wrongly or performing an extraneous harmful action. Use it to enumerate the failure space for each modelled human action.
- **THERP — Technique for Human Error Rate Prediction** (Section 8.5.1, Swain & Guttmann, NUREG/CR-1278): A comprehensive method covering task analysis, error representation, and quantification, built around an HRA event tree with five dependency levels and error-factor uncertainty bounds. Use it for routine, proceduralised tasks under normal conditions — its sweet spot is ground processing and pre/post-flight checks.
- **CREAM — Cognitive Reliability and Error Analysis Method** (Section 8.5.2, Hollnagel, built on the Contextual Control Model): Decomposes work into fifteen generic cognitive activities mapped to four functions (Observation, Interpretation, Planning, Execution), with a basic method for screening and an extended method for detail. Use it for cognitive error analysis and for screening well-understood tasks performed in a new setting.
- **NARA — Nuclear Action Reliability Assessment** (Section 8.5.3, a refinement of HEART): A detailed method that adjusts generic-task basic HEPs using 18 Error Producing Conditions (EPCs). Use it when you want quantification anchored in real human-error data (CORE-DATA) rather than expert judgement, and when only generic task definitions are available.
- **SPAR-H — Standard Plant Analysis Risk HRA Method** (Section 8.5.4, Gertman et al., NUREG/CR-6883): Reduces every task to "diagnosis" and/or "action," adjusts base rates with eight PSFs and a dependency table, and drives the analyst through standard worksheets. Use it when repeatability across analysts matters and when tasks are not yet specified in detail.
- **General Time-Reliability Model / Time Response Curve (TRC)** (Section 8.7.1): A model that converts an available time window and a crew response-time distribution into a HEP for time-critical post-initiator actions. Use it when an action must complete inside a hard time window (e.g., abort, command destruct).
- **Context-based Software Risk Model (CSRM)** (Section 9.4): A risk-modeling framework that represents a software action as the logic intersection of a system entering a condition and the software responding (correctly or not), so it can express both unconditional and conditional, and both recoverable and critical, software failures. Use it to integrate software into an existing PRA at the system functional level. Applied in two stages — Specification-Level and Design-Level.
- **Dynamic Flowgraph Methodology (DFM)** (Section 9.4.4, used inside CSRM): Represents system and control-software variables as nodes joined by cause-effect edges in multi-valued logic and discrete time, then searches for time-dependent "cut-sets" (technically prime implicants) of function failure or success. Use it for complex, time-dependent hardware/software interactions where binary fault trees are inadequate.
- **Software Reliability Growth Models (SRGMs)** (Section 9.4.6.4): Black-box models that estimate how reliability grows as defects are found and removed during test. Use them in Design-Level CSRM to quantify routine "type B" software cut-sets from system-specific debugging data.

## Key Concepts

- **Vulnerability to human error**: How much a system needs HRA depends on its complexity, how much the human interacts with it, and how tightly the human-system is coupled. A tightly coupled system denies the operator alternatives or time to wait for repair, raising the stakes of any error.
- **Performance Shaping Factors (PSFs)**: Context factors — training, stress, available time, ergonomics, working conditions, crew collaboration — that raise or lower the chance an error occurs. PSF lists and the way they multiply or adjust a base HEP are method-specific; THERP names three and lets the analyst add more, CREAM uses nine, SPAR-H uses eight, NARA uses 18 EPCs.
- **Human Error Probability (HEP)**: The quantity HRA ultimately produces for each human basic event, carried into the PRA with an associated uncertainty (typically an error factor on a lognormal distribution).
- **Two-phase HSI quantification**: Screening analysis first reduces the candidate HSIs to the risk-significant ones; detailed analysis is then performed only on what survives. Screening can be qualitative (rule-based exclusion), quantitative (conservative HEPs of roughly 0.1 to 1.0, complete dependence assumed among related actions in the same cut-set), or both.
- **Five-level dependency (THERP)**: Zero, low, moderate, high, and complete dependency between consecutive actions, with time between tasks as the key driver; complete dependency forces the conditional HEP to 1.0. This is how THERP keeps multiple related errors from being treated as falsely independent.
- **Error Factor (EF) and lognormal bounds**: Uncertainty is expressed by multiplying/dividing the median HEP by an EF, with the lognormal as the default distribution because failure probabilities live naturally on a log scale. Larger EFs are conventionally assigned to smaller HEPs, reflecting greater uncertainty at the low end.
- **Software risk is design risk**: In the 1998–2007 decade, software was a critical contributor in roughly 60% of NASA's high-visibility mission failures, and all of those software failures were rooted in design faults — the code met its specification but the specification was wrong for the encountered condition (Mars Global Surveyor, Mars Polar Lander).
- **Defect vs. fault vs. anomaly/failure**: A defect is a deviation from the intended software; a fault is a defect serious enough to cause trouble if executed; an anomaly or failure is the *execution-time* manifestation. A latent defect stays dormant and inconsequential unless its code is actually called upon during the mission.
- **Three defect-introduction types**: Type 1 design/specification defects (requirements and module-spec phases), Type 2 translation/coding defects, and Type 3 parameter/data-entry defects — all of them ultimately human errors in development activities.
- **Conditional vs. unconditional software failure**: Unconditional failures happen spontaneously during a routine function under nominal conditions; conditional failures are tied to a trigger condition that demands a non-routine software response. Traditional black-box failure-rate models capture the former but miss the latter, which dominates the real failure history.
- **Recoverable vs. critical software failure**: A recoverable failure can be restored by built-in fault tolerance (e.g., switch to a redundant unit and run a recovery routine); a critical failure cannot be isolated by fault tolerance and, if uncompensated, leads to mission loss.
- **Condition coverage**: CSRM's organising idea — systematically identifying the mission conditions that demand differentiated software responses, mirroring the orderly design process. This makes the software PRA double as a design validation contribution. It does not mean exhaustive combinatorial coverage of all inputs.
- **Three cut-set types (CSRM)**: Type A non-software cut-sets, Type B "software only" cut-sets (spontaneous failure under routine conditions, quantifiable with a failure-rate model), and Type C "non-software-triggered, software-related" cut-sets (conditional failures triggered by a hardware/external event, quantifiable as a failure-on-demand conditional probability).
- **Specification-Level vs. Design-Level CSRM**: Specification-Level runs early using top-level design info and generic/surrogate data; Design-Level runs later with detailed design info and system-specific test data, and crucially exchanges information with the software test team to "risk-inform" testing.
- **Five basic software-reliability data types**: Surrogate Raw (SR), Surrogate Parametric (SP), System-specific Debugging (SD), System-specific Operational (SO), and Risk-informed Test (RT) data — the last produced *in concurrence with* the CSRM effort rather than independently of it.

## Mental Models

- **HRA is hardware reliability for people**: Just as hardware is classified by failure-on-demand vs. running failure, humans are classified by pre/initiator/post-initiator timing and by skill/rule/knowledge cognition. The classification step is what makes data collection and quantification tractable — it is not bureaucratic overhead.
- **Screening before detail, always**: Both HRA and CSRM funnel a large candidate set through cheap, conservative screening so expensive detailed analysis is spent only on what actually drives the risk. If a conservatively-quantified HSI never appears in a dominant cut-set, stop analysing it.
- **The clock is the model**: For time-critical actions, the HEP *is* the probability that crew response time exceeds the available window. The Cassini Flight Control Officer command-destruct example shows the whole HEP collapsing into one integral over a response-time distribution and a time window — and warns that large windows can drive HEPs absurdly low (below 1E-5), so non-time-dependent execution errors must still be added.
- **"Reliable" software can still be wrong**: Reliability defined as compliance with specification does not equal safety. Most major software-driven mission losses were perfectly compliant with their (flawed) specifications. So software risk analysis must reach *into the design logic*, not stop at code-versus-spec verification.
- **A failure rate is the wrong shape for conditional failures**: A single average failure rate across the operational profile assumes time-driven, semi-random failure. Real critical software failures are deterministic responses to specific trigger conditions — a "failure on demand," like a hardware component's chance of a successful start. Force-fitting a failure rate hides the very failures that matter.
- **The AND-gate dissolves the testing problem**: Because a Type C scenario requires both a trigger condition AND a faulty software response, the conditional software response only needs to be bounded *given* the trigger. The MiniAERCam example shows this turns a seemingly prohibitive test burden into a finite, computable number of targeted samples (e.g., ~1010 successful samples for a 95%-confidence bound on a 1.9E-3 target).
- **Sample the variability, not the same test twice**: Risk-informed test confidence is only valid if the tests genuinely sample the dimensions of variability in the trigger condition (leak rate, spacecraft orientation). Repeating an already-passed test inflates confidence falsely — and the safeguard is engineering judgement, not more statistics.
- **Sub-allocate top-down when data is coarse**: When surrogate data only exists at a high failure-mode level ("failure of continuous control function") but you need lower-level modes, apportion the high-level probability across sub-modes by assumed, modelled, or expert-elicited ratios to get a "good enough" starting point for directed testing.

## Anti-patterns

- **Mis-scoping the human contribution**: The Guide warns that the number and type of errors admitted into the analysis can drive either an under- or over-estimation of the human contribution to system risk — so problem definition must be deliberate, not arbitrary.
- **Assuming software risk is negligible without demonstration**: Many programmes assumed software contributions to mission risk were negligible versus hardware, or applied coarse methods, *without a formal demonstration of safe performance* — a notion the failure data directly contradicts.
- **Relying on V&V alone to catch design errors**: Traditional software V&V is reasonably effective against random coding and data-entry faults but not against design and specification errors, especially in the "gray area" of how software interacts with the balance of system — which is precisely where the major failures occurred.
- **Forcing a failure-rate model onto conditional failures**: Quantifying a Type C conditional failure with an unconditional time-based failure rate is named as a mismatch; these must be quantified as failure-on-demand conditional probabilities.
- **False confidence from redundant testing**: Counting repeated runs of an already-successful test as new evidence generates optimistic estimates and false confidence — an explicitly flagged logic trap.

## Key Takeaways
1. HRA exists to keep the PRA honest about people: identify the human-system interactions, classify them by timing and cognition, screen aggressively, and quantify the survivors as human basic events with HEPs and uncertainty bounds.
2. Error Identification is the pivot of HRA — get the EOM/ECOM enumeration and the PSFs wrong and every downstream number is wrong.
3. No single HRA method is universal: THERP suits granular routine ground tasks, CREAM and NARA offer generic-task screening, SPAR-H buys repeatability, and methods are routinely combined (e.g., CREAM as the general method with THERP for dependency and uncertainty in the Space Shuttle PRA).
4. Time-critical human actions are quantified by comparing a crew response-time distribution against the available time window via a Time Response Curve.
5. Software risk in space systems is overwhelmingly *design* risk — compliant software responding incorrectly to an unanticipated condition — so analysis must validate design logic, not just code-to-spec compliance.
6. CSRM's central move is splitting software risk into unconditional/routine (failure-rate-quantifiable) and conditional/triggered (failure-on-demand-quantifiable) contributions, and identifying which are recoverable versus critical.
7. CSRM integrates with conventional PRA via "entry-point events" that can be expanded by fault trees or DFM and then transferred back as cut-sets, carrying probability and uncertainty into the existing event-tree/fault-tree model.
8. Risk-informed testing is the payoff of CSRM: it prioritises the highest-risk software scenarios, makes the conditional-probability bound a concrete test target, and feeds results back through Bayesian updating — something hardware PRA cannot do because testing actually changes the residual software risk.
