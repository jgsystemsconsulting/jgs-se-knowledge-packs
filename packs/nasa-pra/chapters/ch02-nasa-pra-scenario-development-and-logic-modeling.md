# Chapter — Scenario Development and Logic Modeling (MLD, ESD, Event Trees, Fault Trees)

## Core Idea
Scenario development is the engine that turns the risk-triplet idea (scenario, frequency, consequence) into a quantitative model. Beginning with the entity under assessment — a ground facility, launch vehicle, orbiting asset, or instrument — the PRA team systematically asks "what can go wrong?", traces each upset from its initiating event through the success or failure of intervening systems and human actions to a defined end state, and renders that progression as Boolean logic. Because Boolean algebra applied to a scenario yields the very expression needed to compute its frequency, the modeling artifacts (master logic diagrams, event sequence diagrams, event trees, and fault trees) are not just pictures: each one is a graphical statement of a Boolean equation that, once linked together and populated with reliability data, evaluates to the frequency the risk triplet requires. This chapter is where `nasa-risk`'s qualitative scenario thinking acquires its quantitative spine.

The governing relation is that a specific event-sequence frequency equals the frequency of its initiating event multiplied by the conditional probability of reaching that sequence's end state given the initiator has occurred. Everything in the chapter — familiarization, success criteria, IE identification, tree construction, linking, and reduction — exists to make that product computable and defensible.

## Frameworks Introduced (exact source names, when/how)
- **Risk triplet (scenario / frequency / consequence)** — invoked from Section 2.1 as the conceptual anchor; scenario development is described as fundamental to the entire concept of risk because it is what makes frequency quantification possible.
- **Dependency matrix** — built during the system-analysis portion of a PRA. An explicit table (read column-by-column, the column system being supported by the row systems marked) showing how each system functionally supports the others, so that a failure propagating from one pivotal event into later events is modeled consistently. The source notes a fully developed matrix can carry more than a simple "X" (e.g., endnotes describing support type, or mission-phase-dependent entries).
- **Mission success criteria** — define risk-assessment end states. Must state what the entity has to accomplish to succeed and supply temporal/phase-dependent requirements (operating durations, phase differentiation). Must be mutually exclusive in a logical sense; the source points the reader to the NASA System Safety Handbook §3.1.1 for probabilistic mission-success requirements.
- **System success criteria** — apply to individual systems; include a temporal component and a statement of redundancy. The fault-tree top-event logic is taken as the Boolean complement of the success criterion (success = "at least one of three strings starts and runs 20 min" → top event = "all three fail to start or fail to run 20 min").
- **Master Logic Diagram (MLD)** — a deductive technique for identifying initiating events; the source gives a notional MLD organized around kinetic-energy upsets (performance, impact, atmospheric effects). Used alongside FMEA and fault-tree reasoning to deduce candidate initiators.
- **Event Sequence Diagram (ESD)** — an inductive, success-oriented graphic developed one-per-IE that maps every path from the initiator to its end states; symbol set includes initiation, mitigation block, aggravating block, arrows, connectors, and termination symbols, plus a legend.
- **Event Tree (ET)** — the quantitative inductive graphic, typically one per IE, often derived from a refined ESD; portrays the progression of events over time and is the structure to which fault trees are linked.
- **Fault Tree (FT)** — a deductive logic model in which a system-level failure (the top event) is decomposed backward through gates to basic events whose failure/success probabilities are known. Construction and analysis procedures are referenced to the *Fault Tree Handbook with Aerospace Applications* (NASA, 2002).
- **Linking** — the standard technique of joining ETs and FTs (and ET-to-ET) so the top event of an FT equals the failure of a specific pivotal event in the ET, allowing a Boolean equation to be derived per event sequence.
- **Minimal cut sets (MCSs)** — the minimum subsets of basic events whose simultaneous occurrence forces the top event; the target of Boolean reduction.

## Key Concepts
- **System familiarization** is a prerequisite, not a formality. Because every model is an approximation, the team will deliberately omit nuances — but they must be familiar enough to justify each omission. Available depth scales with system maturity: sparse during conceptual design (elicit from staff), richer at final design, and richest for an operating facility where O&M logs and personnel reveal actual behavior.
- **Two flavors of the risk model.** ETs naturally portray sequential progressions over time (mission phases); FTs naturally capture the failure logic of complex systems. Building a model from FTs or ETs alone is theoretically possible but impractical except in trivial cases, so the two are combined.
- **End states vs. transition states.** An end state terminates a sequence because the outcome relative to mission success criteria is known (e.g., OK, LOC = loss of crew, LOV = loss of vehicle). A transition state hands an under-developed sequence off to another tree for further modeling; every sequence is eventually carried far enough to resolve an end state.
- **Initiating events (IEs).** An IE is a departure from the desired operational envelope to a state requiring control intervention by human or machine. The near-infinite continuum of possible upsets (a rocket failing at t=1.1s, 1.2s, …) is grouped into like categories (e.g., one frequency for failure over t=0–10s) to keep the model tractable.
- **The accident-scenario elements.** A scenario unfolds as IE → control/safety-function response → mitigation; hazards can impinge by acting as enabling events, by affecting IE occurrence, by challenging controls, by defeating mitigating systems, or by failing to limit consequences. Systems are safe because IEs are rare and, even when they occur, control and mitigation rarely both fail.
- **Two IE-development approaches.** Either develop an explicit IE set (e.g., via MLD), or replace the IEs with a single "mission entry point" (often "launch") and model the rest with phased FTs linked to the ET. The single-entry approach risks burying actual upset conditions inside the trees; the source stresses this is acceptable only if hazards and enabling conditions feeding later-phase top events are still modeled holistically.
- **Large-ET vs. large-FT styles.** When repair/recovery is precluded, some analysts favor a large ET; when maintenance or recovery occurs (typical of repairable ground facilities reaching a time-independent availability state), large FTs are preferred. The large-FT method cannot directly tell *when* a failure end state occurs; the large-single-entry-ET method strains on complex repairable systems.
- **Basic event naming.** Names should be informative, encoding hardware item, failure mode, mission phase, and owning system, typically as delimiter-separated fields (e.g., hardware-mode-phase-system). The only hard constraint is compatibility with the PRA software (usually a character limit).
- **FT construction direction and limit of resolution.** Start from the top event (the complement of the system success criterion) and deductively work downward to finer resolution until development below a gate consists only of basic events — component failures, undeveloped faults, phenomenological events, separately-developed support faults, software errors, or human actions. Develop only to where failure data exist or the analysis requires.
- **FT modeling refinements.** House events act as on/off switches (probability fixed at 0 or 1, no data needed) and often encode conditional dependencies. Passive/dormant components are frequently excluded unless they are single-failure points or have comparable likelihood. Test/maintenance unavailability and component fault are mutually exclusive — reduction must drop impossible cut sets combining them. Pre-accident and certain post-accident human errors belong in the FT (other recovery actions are handled by HRA, Chapter 7); dependent failures route to Chapter 8; software errors must be incorporated because a catastrophic software fault can defeat all mechanical redundancy.
- **Reduction then quantification.** FT evaluation is two steps: Boolean reduction to MCSs, then quantification of the top-event probability from basic-event probabilities. The rare-event approximation may be used by software, and conditional probabilities may be ignored unless explicitly modeled — so logic must be kept compatible with the tool's approximations.

## Mental Models
- **Read the dependency matrix by column to trace support.** To know what holds up a given system, scan down its column for the X-marked rows. This same reading is what keeps a failure originating in one pivotal event from being silently lost when it should cascade into later events.
- **Success criteria are the two switches that set the model's targets.** Mission success criteria fix the event-sequence *end states*; system success criteria fix the FT *top events*. Convert any system success statement into its Boolean complement and you have the top event to decompose.
- **ESD first, ET second.** Sketch the success-oriented ESD (IE → anticipated response → backup systems/human actions → end states for the paths that fail), because its looser morphology displays complex IE-response relationships more readily; then abridge it into the more rigid, quantitative ET. The ESD doubles as the place to document the assumptions and justify why ET simplifications are conservative.
- **Treat a linking sequence as a new initiator.** When one ET's sequence links into another tree, view that sequence as the IE of the second tree; the linked Boolean equation is just the conjunction of the upstream sequence's expression with each downstream sequence's expression (concatenated names like W5-IE1 track the join).
- **Build fault trees backward from the failure, not forward from parts.** Postulate the top event and chase causes downward in order of decreasing generality (subsystems → components → software/human) until you bottom out in basic events with known data. Stop at the limit of resolution, not at exhaustive detail.
- **Two cheap-data shortcuts.** A pivotal event with good probabilistic data from similar missions or testing needs no FT — assign the data directly and let it behave as a basic event. Likewise, group initiators into categories rather than enumerating every instant of possible failure.

## Anti-patterns
- **Compartmentalizing failures behind a "launch" single-entry point.** The source warns that when the model is decomposed into discrete phases under a single "launch" initiator without coupling the phased top events back to the hazards and failure causes that actually represent the IEs, the scenario concept gets discarded and failures become compartmentalized — a modeling failure attributable to the missing coupling, not to using "launch" per se.
- **Letting impossible cut sets survive reduction.** Because a component fault and that component's test/maintenance unavailability are mutually exclusive, cut sets that combine them are physically impossible and must be excluded during FT reduction.
- **Running logic that the software cannot honor.** Since some PRA tools ignore conditional probabilities unless expressly modeled and apply the rare-event approximation to unions, the source cautions that logic models must be kept compatible with every approximation built into the software algorithms.

## Key Takeaways
1. A scenario's frequency is the IE frequency times the conditional end-state probability; scenario development exists to make that product computable, which is exactly what deepens `nasa-risk`'s qualitative scenarios into quantitative PRA.
2. ESDs, ETs, and FTs are all graphical Boolean equations; the power comes from linking them so a single Boolean expression per event sequence can be reduced to minimal cut sets and quantified with reliability data.
3. Mission success criteria define end states (ET outputs); system success criteria define FT top events via their Boolean complement — get these wrong and the whole logic model points at the wrong targets.
4. IEs are departures from the operational envelope, identified deductively (MLD, FMEA, fault-tree reasoning) and grouped into categories; the choice between an explicit IE set and a single mission entry point shapes how hazards must be re-coupled into the model.
5. Pick the tree style to match the system: large ETs when no repair is possible, large FTs for repairable facilities that settle into a time-independent availability — accepting that large FTs cannot say *when* an end state occurs.
6. Fault trees are built top-down to a deliberate limit of resolution, with disciplined handling of house events, passive components, test/maintenance exclusivity, human errors, and software faults, then reduced to MCSs before quantification.
