# Chapter 7: Software Reliability

> Source: FAA-HDBK-006D (2020), Appendix C, "Software Reliability." This pack uses the 006D revision, not 006B.

## Core Idea
Software reliability is the capacity of a software system to deliver its required functions, under stated conditions, for a stated period. In FAA-HDBK-006D it is treated as a discipline that *complements* — and runs alongside — the hardware-oriented RMA activities covered elsewhere in the handbook. The handbook's central move is to split the work along the acquisition relationship: the **Government** (the FAA, which starts the project) needs ways to predict software quality *before* a vendor is chosen or any code exists, while the **developer/contractor** needs a reliability-growth program that improves the software as it is built and operated. Crucially, software does not behave like hardware: it has no wear-out phase, and every new feature or fix temporarily *raises* the failure count rather than lowering it.

## Frameworks Introduced
- **The software reliability process (plan → analyze → allocate → test → monitor → correct)** — the handbook's generic activity flow. It opens with planning (risk assessment, data collection, tool selection), proceeds to analysis that surfaces potential failure modes, allocates requirements to suppress those modes and raise reliability, modifies and runs test procedures to catch the faults, and continues monitoring and correction into operational use.
  - When/how to use: as the spine of any software reliability effort, for both Government and contractor activities described in C.2.
- **Software analogues of the Section 7 RMA tools** — software defect root cause analysis, software FMEA, and software fault tree analysis.
  - When/how to use: during the analysis step to detect failure modes. The handbook is explicit that these are conceptually similar to the hardware root cause analysis, FMEA, and Fault Tree Analysis in Section 7, but their *execution differs* — do not assume the hardware procedure transfers unchanged.
- **Government early software reliability prediction** — a methodology that estimates software quality before contract solicitation by drawing on the relationship between quality and the development organization's maturity level, domain knowledge, and process activities.
  - When/how to use: before a development vendor is selected and before code is written, so the FAA can improve software quality from the solicitation onward. Originated at the US Air Force Rome Laboratories in 1992 (for ADA weapons systems) and later adopted by the Nuclear Regulatory Commission; it is a best practice in IEEE Std 1633-2016, clauses 5.3 and 6.2 and Annex B. It works with COTS software as well, and because it calculates a defect density (defects per line of code), it needs a software-lines-of-code (SLOC) estimate.
- **IEEE Std 1633-2016 prediction model catalog (Table C-1)** — a menu of software reliability prediction models, split into *normative* models (the first three rows, formally approved by the standard) and *informative* models (the remaining, optional rows). Listed models include the Shortcut and Full-Scale models (Neufelder), metric-based models (Smidts), the Rayleigh/Putnam historical-data model, RADC TR-92-52 / SAIC (noted as obsolete and aircraft-oriented), and the Neufelder/Quanterion model. Most produce a defect-density output.
  - When/how to use: to choose a prediction model by trading off number of inputs, effort, year developed, and relative accuracy.
- **Neufelder Full-Scale Model (forms A / B / C)** — a questionnaire-based predictor, recommended (per Neufelder, 2000) for predicting software defect density during the proposal phase. Form A has 94 questions answerable from the software development plan or team interviews; form B adds 132 questions requiring plan and schedule knowledge (226 total); form C adds 151 more covering requirements, design, and test strategy (377 total).
  - When/how to use: in solicitation/proposal-phase prediction, trade-off analysis of reliability activities, and (form C) improving the effectiveness of development deliverables. The handbook recommends it because it has been cross-referenced to RTCA DO-178C activities (and could be cross-referenced to DO-278A), involves 100–300 questions answered by development personnel, supports trade-off analysis, and may suit smaller projects.
- **Question categories correlated to fielded defects (Table C-3)** — the Full-Scale Model's questions grouped into categories such as avoiding "big blobs" (small increments, more granular tasks, shorter gaps between reviews, smaller and more frequent releases), change management, coding practices, defect reduction techniques/tools, defect tracking, design practices, domain expertise, execution of the project, inherent risks, personnel organization/experience, and planning.
  - When/how to use: to understand *which* development behaviors the prediction is sensitive to, and therefore which behaviors to improve.
- **Software reliability growth program** — the developer/contractor counterpart. It shifts the headline metric away from Mean Time Between Failure (MTBF) toward problem-report metrics, with the objective of catching and correcting defects earlier in the lifecycle.
  - When/how to use: by the developer (often a Government contractor) implementing custom software where a COTS product is unsuitable.
- **Problem report metrics** — typically a sequence of failure times or other events, used to measure time between events, quantify the improvement from removing errors, decide when to release or update a version, and assess software/system stability.
  - When/how to use: throughout development as the operational measure of reliability growth.
- **Software fault taxonomies** — schemes that sort a defect into classes pointing at the part of the development process needing investigation; they also organize data for a software FMEA. Many variations exist, but activities are broadly divided into design, code, and test.
  - When/how to use: to feed and structure software FMEA and to localize where in the process a defect originated.

## Key Concepts
- **Software has no wear-out phase.** Over a software system's life, failures stem from design (logical errors) and from coding errors. Failures fall as defects are removed through testing and correction — but introducing new features or fixes temporarily increases failures. Late in system life, modification tends not to be worth the effort.
- **Software reliability complements, not replaces, RMA.** The handbook positions software reliability engineering as support for high-reliability software development that sits *alongside* the RMA activities it presents — not as a substitute for them.
- **Two distinct lanes — acquire vs. build.** Because the FAA initiates the project, many decisions are locked before a vendor is chosen; hence the Government's emphasis on *early prediction*. The developer, by contrast, runs a *growth* program once it is building software the FAA could not buy off the shelf.
- **Defect density as the common currency.** The prediction models, and the early-prediction methodology, express results as defect density (defects per line of code), which is why a SLOC estimate is a prerequisite.
- **Normative vs. informative models.** In IEEE Std 1633-2016's catalog, only the first three models are normative (approved); the rest are informative options. (Note: the bracketed references inside Table C-1 are internal to the IEEE standard, not to FAA-HDBK-006D.)
- **Intellectual property reality.** Early SRE tools and data developed by the Government are public domain; later private-sector work tends to be proprietary, so adopting current tools and data carries cost to the FAA.
- **From MTBF to problem reports.** In the growth program the operative metric deliberately moves off MTBF toward problem-report data, reflecting a focus on early defect correction rather than time-to-failure.
- **Implementation prerequisites for the FAA.** Standing up Government early prediction requires building a cadre with software reliability engineering expertise (and RTCA DO-278A development expertise), and the method works best after the model is validated against data from past FAA software efforts.
- **It does not replace mandated development work.** Early prediction leverages DO-278A safety-risk-management work but does not substitute for the software development activities and requirements imposed by AMS policy (para 4.12) and RTCA DO-278A.

## Mental Models
- **"Aim small, miss small."** The handbook's own framing for the highest-leverage development behavior category: keep increments small, tasks granular, the gap between reviews short, and releases small and frequent. Smaller scope per unit of work means smaller, more catchable defects.
- **Prediction is a pre-contract instrument, growth is a build-time instrument.** If you have not yet picked a vendor, your lever is *prediction* (estimate quality from organizational maturity and process). Once code is being written, your lever is *growth* (problem-report metrics driving earlier correction). Don't reach for the wrong instrument for the phase you are in.
- **Conceptual similarity is not procedural similarity.** Software root cause analysis, FMEA, and fault tree analysis share the *idea* of their Section 7 hardware namesakes but are executed differently — treat the name as a pointer, not a procedure.
- **Choose a model by its trade-off shape.** Each prediction model trades number of inputs against effort and accuracy (and has an era and a defect-density output); pick by what you can afford to feed it and how much accuracy the decision needs.
- **More questions, more confidence — at a cost.** Moving from Full-Scale form A (94) to B (226) to C (377) buys deeper prediction and sensitivity analysis, but each step demands access to more development artifacts (plans, schedules, then requirements/design/test strategy).

## Anti-patterns
- **Treating software like hardware ("just swap in a better part").** The handbook explicitly contrasts software with hardware: hardware is often improved by new equipment, whereas adding software features or fixes temporarily *increases* failures, and software never reaches a wear-out phase. Expecting hardware-style improvement curves is a category error.
- **Modifying software at end of life.** The source notes that, as a system nears retirement, reworking its software rarely repays the effort — late churn buys little.
- **Assuming early prediction substitutes for required development work.** The handbook warns that early software reliability prediction does not replace the development activities and requirements mandated by AMS policy para 4.12 and RTCA DO-278A.

## Key Takeaways
1. Software reliability is the ability to perform required functions under stated conditions for a stated time, and it **complements** the handbook's hardware RMA activities rather than replacing them.
2. Software differs fundamentally from hardware: **no wear-out phase**, and new features/fixes **temporarily raise** failure counts.
3. The work splits into two lanes — **Government early prediction** (before vendor selection / before code) and **developer reliability growth** (during build) — reflecting the acquisition relationship.
4. Early prediction is rooted in **organizational maturity, domain knowledge, and process activities**; it originated at USAF Rome Laboratories (1992), is a best practice in **IEEE Std 1633-2016**, and outputs **defect density** (needing a SLOC estimate).
5. The **Neufelder Full-Scale Model** (forms A/B/C: 94 / 226 / 377 questions) is recommended for proposal-phase prediction, partly because it cross-references **RTCA DO-178C** activities.
6. The growth program shifts the headline metric from **MTBF to problem-report metrics**, aiming to correct defects earlier; **software fault taxonomies** (design/code/test) organize data for a software FMEA.
7. Implementation requires the FAA to build **SRE and DO-278A expertise**, validate the model against **past FAA data**, and recognize that prediction **does not substitute** for mandated DO-278A / AMS development work.

- **Section 7 RMA tools (this handbook)**: software root cause analysis, software FMEA, and software fault tree analysis are the software analogues of the hardware root cause analysis, FMEA, and Fault Tree Analysis — same concepts, different execution.
- **RTCA DO-178C and DO-278A**: the airborne and ground/CNS-ATM software assurance standards the Full-Scale Model cross-references and that early prediction leverages and must not replace.
- **IEEE Std 1633-2016 (Recommended Practice on Software Reliability)**: the source of the early-prediction best practice (clauses 5.3, 6.2, Annex B) and the prediction-model catalog in Table C-1.
