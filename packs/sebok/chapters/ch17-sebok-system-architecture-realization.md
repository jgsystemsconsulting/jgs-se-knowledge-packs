# Chapter 17 — System Architecture Design Definition, Realization and Deployment

## Core Idea
This chapter spans the arc from defining a system's architecture, through detailing its design, to building, verifying, validating, transitioning, and operating it. **System Architecture Design Definition** establishes the behavior and structure characteristics of a system in accordance with derived requirements, so that the system elements work together in their operating environment to satisfy stakeholder needs — it is SEBoK's implementation of the ISO/IEC/IEEE 15288 *Architecture Definition* process. **System Detailed Design Definition** then supplements the architecture with the design characteristics and enablers needed for implementation. **System Realization** creates and tests versions of the system (implementation, integration, verification, validation), and **System Deployment and Use** transitions the verified system into operation and sustains it. The through-line is the Vee: design definition descends the left side, realization climbs the right side, and verification and validation are performed *concurrently and recursively* level by level — never as a single big bang at the end.

## Frameworks Introduced (exact SEBoK terms)
- **System Architecture Design Definition** — the knowledge area (and SEBoK's implementation of the *Architecture Definition* technical-lifecycle process from ISO/IEC/IEEE 15288) that defines structural elements and their behavior interactions in the operating environment to satisfy stakeholder needs. Use it early in the life cycle to translate concept into a viable design; it decomposes into **Functional Architecture**, **Logical Architecture**, and **Physical Architecture** sub-articles.
- **Functional Architecture** — an architectural *view* (not a standalone sub-unit), defined as the inter-related set of transformative processes and input-output tasks (functions) a system performs to produce outputs supporting mission objectives. Use it to capture what a system can do and how those actions relate; in SysML it is expressed jointly with the behavioral view via use case, block definition, activity, and state machine diagrams.
- **Logical Architecture** — a contraction of *Logical View of the System Architecture*; the elaboration of functionality and behavior of the future system as it should operate in service. Use *Logical Architecture Model Development* to produce three views — the **functional architecture view**, the **behavioral architecture view**, plus a **temporal architecture view** — kept (as far as possible) independent of implementation, yielding the **independent logical architecture model**.
- **Physical Architecture** — a contraction of *Physical View of the System Architecture*; an arrangement of concrete physical elements — system elements plus the physical interfaces binding them — that delivers the solution and satisfies the logical architecture and system requirements per ISO/IEC/IEEE 26702. Use *Physical Architecture Model Development* to allocate logical elements to **system elements**, group them into **system element groups** (subsystems), and select the preferred candidate.
- **Platform Independent Model [PIM] / Platform Specific Model [PSM]** — the model-based pairing SEBoK uses for the logical-to-physical step. The PIM identifies logical configuration items without a specific design implementation (with allocated functions, attributes, interfaces, performance); the PSM identifies a specific design implementation with a manufacturer's part number. Use them to separate "what the element is" from "which exact technology realizes it" (e.g., a logical combustion engine vs. a specific V8 gas engine).
- **Authoritative Source of Truth [ASoT]** — the digital system design model serving as the single repository of a project's approved technical baseline (requirements, functional, logical, physical) across each abstraction level. Use it for collaboration, decision making, and product-line reuse libraries.
- **Digital thread / Digital twin** — the analytical frameworks and authoritative representations the model-based approach is built around. A *digital thread* provides end-to-end simulation representations exchanging information across modeling tools through the life cycle; a *digital twin* is an authoritative representation of the physical system with the data, models, and infrastructure to optimize the design digitally. Use them to verify design decisions and discover defects before producing physical prototypes.
- **System Detailed Design Definition** — the process that develops, documents, and communicates realization of the architecture through a complete set of **design characteristics** (dimensions, shapes, materials, data structures) and **design enablers** (equations, drawings, diagrams, patterns, algorithms, heuristics). Use it as the "how-" or "implement-to" link between architecture and the implementation of technological system elements.
- **System Analysis** — the process that performs objective quantitative assessments — **trade-off studies** built from **cost analysis**, **technical risks analysis**, and **effectiveness analysis** — to select and update the most efficient architecture. Use it whenever technical choices or decisions are made; it supports the Logical and Physical Architecture Model Development processes and pairs with **Decision Management**.
- **System Realization** — the knowledge area grouping the activities that create and test versions of the system: **System Implementation**, **System Integration**, **System Verification**, and **System Validation**, performed concurrently, iteratively, and recursively per the chosen life cycle model. DAU and NASA additionally fold **transition** into realization.
- **System Implementation** — the process that yields the lowest-level system elements (made, bought, or reused) conforming to design properties/requirements via fabrication, coding, or operational-procedure development. Use it to bridge system definition and integration.
- **System Integration** — the process that iteratively combines implemented system elements into **aggregates** and full or partial configurations, applying V&V actions during assembly to confirm the elements function as a whole. Use it as the bottom-up branch of the Vee; it deals only with developmental items, not production-line assembly.
- **System Verification** — confirmation through objective evidence that specified requirements, architecture, and design characteristics have been fulfilled ("built right"). Use it transversally throughout the life cycle via a justified **verification strategy** and techniques such as inspection, analysis, analogy/similarity, demonstration, test, and sampling.
- **System Validation** — confirmation through objective evidence that requirements for a specific intended use have been fulfilled ("does right"), ratified by stakeholders. Use it level by level, culminating in **final validation** (integrated system, industrial environment) and **operational validation** (operational environment).
- **System Transition** — the process (under System Deployment and Use) that, per ISO/IEC/IEEE 15288, installs a verified system alongside its relevant enabling systems, bridging qualification and use (handoff to logistics, operations, maintenance, support). Use it to plan installation, checkout, integration, on-site/field testing, run-in, and phasing-in/phasing-out.
- **System Operation** — the process that assigns and trains personnel to operate the system and monitors operator-system and system performance, sustaining mission and business functions and handling eventual decommissioning/disposal.

## Key Concepts
- **Architecture defines behavior *and* structure against derived requirements.** Following Hamilton's Principle (per SEBoK), a system is built from interacting elements that exchange data, energy, or mass to change one another's states, producing continuous, discrete, or emergent behaviors across levels of aggregation/decomposition.
- **The model-based approach replaces disparate documents.** Traditional practice manually organized configuration information in documents using ad-hoc notation without formal semantics, forcing manual synchronization and causing consistency, cost, and schedule problems. The model-based approach uses SysML's standardized graphical/textual notation with formal semantics, with traceability from requirement to functional/logical/physical representations across every abstraction level.
- **Five model-based activities.** Provide a project overview and boundaries; define system requirements and verification criteria (black-box use cases, derived capabilities); define the functional hierarchy and behavior interactions (input-to-output transformations satisfying allocated requirements); define logical structural elements (PIM, white-box analysis, make-vs-buy); define physical structural elements (PSM, recursive EE/ME/SW design with specific part numbers).
- **A function is an input-output transformation.** In functional/logical architecture a function is closer to the mathematical sense — y = f(x, t) — than to the broader SE sense of "actions in pursuit of an outcome." Functions are deduced from functional and interface requirements, or derived from the physical solution and dependent on technology choices.
- **Functional vs. behavioral architecture are distinct lenses.** Functional architecture concerns input-output transformations; behavioral architecture concerns the *sequencing and execution* of actions and how they interact with checkpoint conditions, control flows, and the initiation/completion of other tasks. SysML expresses them jointly.
- **Logical architecture has three model layers.** The **functional architecture model** (functions and sub-functions and their flows), the **behavioral architecture model** (execution sequencing, control/trigger flows, **scenarios of functions**, **operational modes**, and **behavioral patterns** including basic constructs, complex patterns, and FDIR patterns), and the **temporal architecture model** (classification of functions by execution frequency — synchronous/cyclical vs. asynchronous, with a temporal and decisional hierarchy that separates disturbances on high frequencies from changes on low frequencies).
- **Functional decomposition is a static view that needs scenarios.** A head-of-hierarchy function (F0, the mission/"black box") breaks into sub-functions and leaf-functions, but a static hierarchy alone does not show how flows are exchanged; it should be established only once scenarios exist and the logical architecture is close to complete.
- **Physical architecture is driven more by non-functional requirements than by functions.** There are often many physical ways to perform a function but fewer ways to satisfy non-functional requirements (performance, safety, security, environment). A **design property** is created by assigning non-functional requirements, estimates, analyses, or by reusing an existing element.
- **Allocation, partitioning, and grouping.** Logical functions are allocated to **system elements** that can perform them; **partitioning and allocation** use affinity criteria (same technology, similar efficiency, same flow types, centralized/distributed control, close frequency, dependability, environment resistance) to find feasible elements; numerous elements are then grouped into **system element groups** (subsystems). A guideline of five-plus-or-minus-two elements per level keeps definition manageable.
- **Interface standards can be a primary physical driver.** Especially when multiple systems share a common physical architecture, interface standards may be mandated at a high level or derived during development — enabling outcomes such as interoperability, technology insertion, "open systems," and families of systems.
- **Legacy and SoS context shape the physical architecture.** Weak interactions can be handled as static external interface requirements/constraints with design assurance; intense interactions push the work into an enterprise / system of systems approach, requiring open or configurable interfaces and **late binding** of component systems.
- **Design ≠ architecture, but they intertwine.** The trend is to treat them as separate but concurrent, strongly intertwined activities. Design provides the implement-to detail; it gives feedback to the parent architecture to consolidate allocation and partitioning, and **holistic design** considers the system as an interconnected whole within something larger, including environmental impact.
- **Design definition manages obsolescence and make/buy from the start.** It plans technology management, identifies elements at risk of obsolescence, assesses COTS/NDI/reused vs. developed alternatives, and maintains traceability and the design baseline for configuration management.
- **System analysis is a rigorous, iterative decision basis.** It defines assessment criteria from requirements, assesses each candidate's design properties, scores and justifies, and decides — using physical, representation, and analytical models (deterministic, probabilistic/stochastic, multi-criteria) appropriate to project progress, with specialist expertise where criteria are subjective. **Life cycle cost (LCC) / total ownership cost (TOC)** spans development, manufacturing, sales, utilization, supply chain, maintenance, and disposal.
- **Realization maps cleanly onto the Vee.** The left side develops specifications, design descriptions, and V&V plans (flow-down requirements for lower levels); the right side assembles elements (integration) and runs V&V against stakeholder requirements, system requirements, and design properties — bottom-up, continuously/incrementally/iteratively.
- **Implementation produces evidence, packaging, and supporting documentation.** It defines an implementation strategy (fabrication/coding procedures, tools, tolerances, configuration audit criteria), realizes the element to safety/security/privacy/environmental standards, records compliance evidence and **measured properties**, and handles PHS&T (packaging, handling, storage, and transportation) plus operation/maintenance/installation manuals used in deployment.
- **Integration is assembly *plus* verification, minus final validation.** Each implemented element is individually verified and validated before integration; integration adds verification during assembly; the **final validation** (operational tests) authorizes transition for use or production. Integration uses an **integration strategy/plan** built from the V&V strategy, with **aggregates**, assembly tools/procedures, and V&V tools (simulators, stubs/caps, launchers, harnesses, measuring devices).
- **Integration techniques are a trade, usually mixed.** SEBoK names **Global (big-bang)**, **with the stream**, **incremental**, **subsets (functional chains)**, **top-down**, **bottom-up**, and **criterion-driven** integration — each with strengths/weaknesses selected by element type, delivery order, risk, and constraints. The **coupling matrix / N-squared diagram** optimizes aggregate definition and aids fault localization.
- **Verify ≠ validate.** Verification is supplier-oriented, detail/local, glass-box, binary comparison, "done right," referenced to the design — performed by the design team. Validation is end-user-oriented, global, black-box, may need a value judgement, "does right," referenced to system and stakeholder requirements — performed with experts external to the design team, and presupposes verification was done. Verification comes first; validation second.
- **Both V&V are optimized as one strategy, level by level.** A single global validation on a complex integrated system is infeasible (causes of non-conformance can't be isolated), so each system/element is verified, validated, and corrected before integrating into its parent. The combined integration + verification + validation strategy minimizes actions while maximizing defects found and confidence gained, weighing the risk of any excluded check.
- **Transition includes acceptance testing and run-in.** Activities include installation, integration, and V&V in the operational setting; **On-site Acceptance Test (OSAT)**, **Field Acceptance Test**, and **Operational Test and Evaluation (OT&E)**; a **system run-in** period (typically one-to-two years) to confirm reliability/maintainability and integrated logistics support; and **phasing-in/phasing-out** to minimize disruption when replacing legacy systems. **Reliability demonstration testing (RDT)** may be required to meet contractual R&M guarantees.
- **Operation sustains effectiveness and plans the end.** SE monitors measures of operational effectiveness, runs **FRACA** (failure reporting and corrective actions), updates training/certification, manages migration off legacy systems without service breakdown, and plans **service life extension (SLEP)** for end-of-life systems facing **DMSMS** (diminishing manufacturing sources and material shortages), through to disposal compliant with laws and regulations.

## Mental Models
- **Vee as the organizing skeleton.** Definition descends (decomposition, allocation, specs, V&V plans); realization ascends (implementation at the bottom, integration upward). Verification and validation are not a phase at the top — they are woven into both sides, performed inside each level and transverse between levels. Picture V&V actions stacked at every layer, not pooled at the end.
- **PIM → PSM as the logical-to-physical bridge.** First say *what the element is and must do* (logical, technology-independent, contractually baselineable); then say *which exact technology realizes it* (physical, part-numbered). The automobile makes it concrete: a logical "combustion engine" with requirements, then a specific V8 gas/diesel/propane engine with pistons, plugs, and cam shaft.
- **Three logical lenses on one system.** Functional (what transformations), behavioral (in what sequence, triggered how), temporal (at what frequency, decided at which level). A complete logical architecture needs all three; mixing governance/management/operations into one undifferentiated view is a known pitfall.
- **Architecture is the skeleton; design is the flesh.** Architecture sets behavior, structure, and allocation; design supplies the implement-to characteristics and enablers per discipline. They run concurrently and feed each other — discipline engineers detail elements and feed knowledge back to architects, and architects feed allocation feedback down.
- **Non-functional requirements steer physical choices.** When many physical solutions can perform the same function, the deciding vote goes to performance, safety, security, environment, modularity, and resilience. Architect to interfaces and modularity (maximum internal consistency, minimum external interfaces), not just to elements.
- **"Built right" vs. "does right."** Verification asks whether you solved the problem right (objective, against the design); validation asks whether you solved the right problem (judgement, against stakeholder intent). Both use the same techniques (inspection, analysis, demonstration, test, simulation) but for different purposes and references.
- **Model as the authoritative source, simulation as the cheap proving ground.** Build a high-fidelity digital design model (ASoT) and run digital-thread, digital-twin simulations to find and fix defects before the expense of physical prototypes — the governing principle of the model-based approach.
- **Optimization is not a sum of optimized parts.** Optimizing the whole system-of-interest globally is not achieved by summing locally optimized elements — a named system-analysis pitfall and a core systems-thinking caution.

## Anti-patterns
*(Issues SEBoK names explicitly in this knowledge area.)*
- **Decomposition too deep** (logical architecture) — decomposing functions too far, or carrying too many functions/flows in a single block's scenarios or functional model.
- **Not considering inputs and outputs together with functions** — decomposing actions while forgetting or deferring their inputs/outputs, which are integral parts of a function.
- **Considering static decomposition of functions only** — building the static function tree before scenarios exist; the static decomposition should come *after* scenarios and a near-complete logical architecture.
- **Mixing governance, management, and operation** — conflating strategic monitoring, tactical monitoring, and basic operations instead of treating behavioral and temporal architecture models properly.
- **No logical architecture model** (physical architecture) — jumping straight from system requirements to a physical model (common with repeating products), which breaks when a function's input-output domain set changes.
- **Too many levels in a single system block** — a physical architecture block should contain a single level of systems/system elements.
- **Direct allocation on technologies** — allocating high-level functions straight onto hardware/software without alternating logical and physical levels first.
- **Analytical modeling treated as a decision tool** — it is an aid, not the decision; results are only as good as the data and the operational field they apply to.
- **Big-bang integration when inappropriate** — it localizes faults late; verify interfaces progressively. Planning the integration too late (when first elements arrive) is the related scheduling pitfall.
- **Confusing verification and validation** — leads to the wrong reference/baseline and the wrong granularity (detail for verification, global for validation).
- **No verification strategy / using only testing / stopping when funding runs out** — overlooking checks for lack of a justified strategy; relying solely on testing (which only works once implemented) instead of earlier analysis/inspection; and ending V&V by budget exhaustion rather than coverage criteria. The validation mirror: **starting validation at the end of the project** instead of as early as the first requirements are known.

## Key Takeaways
1. **System Architecture Design Definition** defines behavior and structure against derived requirements and is SEBoK's implementation of the ISO/IEC/IEEE 15288 *Architecture Definition* process, decomposing into **Functional**, **Logical**, and **Physical Architecture** views.
2. The chapter's model-based thread uses **SysML** to build an **ASoT** digital design model, with **digital threads** and **digital twins** enabling simulation-based verification before physical prototypes — moving from **PIM** (logical) to **PSM** (physical).
3. **Logical architecture** carries three models — **functional**, **behavioral**, and **temporal** — and must be scenario-driven, not built from a static function tree alone; it aims for an **independent logical architecture model**.
4. **Physical architecture** allocates logical elements to **system elements**, groups them into **system element groups**, and is driven more by **non-functional requirements** than by functions; **interface standards** and **legacy/SoS** context are major shaping forces.
5. **System Detailed Design Definition** supplies **design characteristics** and **design enablers** as the implement-to link between architecture and implementation, managing make/buy and obsolescence and keeping the design baseline traceable.
6. **System Analysis** drives selection via **trade-off studies** of **cost**, **technical risk**, and **effectiveness**, using models matched to project maturity — remembering that global optimization is not a sum of optimized parts.
7. **System Realization** = **Implementation + Integration + Verification + Validation**, performed concurrently/recursively along the Vee; integration is **assembly plus verification minus final validation**, using a strategy of mixed techniques and the coupling-matrix/N-squared method.
8. **Verification ("built right") and validation ("does right")** are distinct but complementary, optimized as one **level-by-level** strategy that minimizes actions while maximizing defects found; **final** and **operational validation** close out the right side of the Vee.
9. **System Transition** installs the verified system (with enabling systems), runs OSAT/Field/OT&E acceptance tests, a **run-in**, and **phasing-in/phasing-out**; **System Operation** sustains effectiveness via **FRACA**, training, migration, **SLEP/DMSMS** handling, and compliant disposal.

## Connects To
- **System Definition** — supplies the requirements, logical/physical architecture, and design descriptions that realization implements, verifies, and validates against; architecture and design definition sit within this part of the life cycle.
- **System Concept Definition (Ch. 16)** — runs concurrently with System Architecture Design Definition; the integrated set of needs and candidate solution classes feed architecture, and the needs become the baseline for **System Validation**.
- **Decision Management** — used together with **System Analysis** to perform the trades and select preferred logical and physical architecture candidates.
- **Risk Management** — **technical risks analysis** within System Analysis (and the integration **criterion-driven** technique keyed to risk) distinguishes technical risk from project risk while sharing management methods.
- **Applying Life Cycle Processes / Life Cycle Models** — determines how concurrently/iteratively/recursively the realization and V&V activities are performed; the Vee, incremental, and evolutionary models all change the emphasis.
- **Product and Service Life Management** — owns sustainment, decommissioning, disposal/retirement, **SLEP**, and **DMSMS** referenced by System Operation and Transition.
- **Logistics / Maintenance** — receive the transition handoff (integrated logistics support, R&M deliverables, support strategy) and own sustainment treated separately from transition.
- **Enterprise Systems Engineering / Systems of Systems** — where intense legacy interactions and late binding of component systems move the work, and where **Enterprise Architecture Frameworks & Methodologies** (e.g., TOGAF, Zachman, DoDAF) supplement domain views.
- **Model-Based Systems Engineering (MBSE)** — the backbone of the model-based architecture approach (SysML, ASoT, digital thread/twin) recommended for quality architecture models.
- **Systems Engineering and Quality Attributes** — home for the non-functional properties (reliability, availability, maintainability, security, etc.) that become **design properties** and dominate physical architecture selection.
- **Standards & references:** **ISO/IEC/IEEE 15288:2023** (System Life Cycle Processes), **ISO/IEC/IEEE 42010** (Architecture Description), **ISO/IEC/IEEE 26702 / ISO/IEC 26702:2007** (Application and Management of the SE Process), **OMG SysML v1.7 and v2 beta** and the **Model Driven Architecture Guide** (PIM/PSM), **INCOSE SE Handbook v5.0**, **NASA SE Handbook (NASA/SP-2007-6105)**, **DAU Defense Acquisition Guidebook**, **ANSI/EIA-632**, and **IEEE-1012** (Verification and Validation).
