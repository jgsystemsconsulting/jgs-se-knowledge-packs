# Chapter — Software Classifications

> Source caveat: This chapter is grounded in NASA Procedural Requirement NPR 7150.2D, a NASA directive rendered from its NODIS (NASA Online Directives Information System) HTML rendition. NPR 7150.2D states software-engineering requirements that complement the broader software-for-systems area. The printed text is uncontrolled; the authoritative version lives in the NODIS Library. All claims below are paraphrased from Appendix D of that directive.

## Core Idea

NPR 7150.2D does not apply its requirements uniformly to every piece of software NASA touches. Instead, it routes each system or subsystem into one of six named classes, and the class determines how many of the directive's requirements apply and how much rigor each carries. Lower classes have their applicable requirement set and associated rigor scaled back. The mechanism that performs this scaling is the Requirements Mapping Matrix, and the classes are the input to it.

The classification is deliberately driven by consequence and dependency, not by language, size, or where the code happens to run. Five underlying factors decide where software lands: how the software is used with or within a NASA system, how critical the parent system is to NASA's major programs and projects, how much humans depend on it, how complex it is to develop and operate, and how large the Agency's investment is. A spreadsheet that flies aboard a crewed vehicle but does nothing for the mission is treated very differently from ground software that calls a launch abort — the factors, not the venue, govern.

## Frameworks Introduced (exact source names, when/how)

- **NASA-wide software classification structure** — the governing taxonomy of Appendix D. It is the structure that decides whether and how the directive's requirements bind a given system or subsystem containing software.
- **Classes A through E** — the engineering-related software classes. They form a graded ladder from human-rated space systems (A) down through design-concept and general-purpose research software (E).
- **Class F** — a separate track named *Business and Information Technology Infrastructure Software*, sitting outside the A-through-E engineering ladder.
- **Requirements Mapping Matrix** — the named instrument that, given a class, scales the number of applicable requirements and their rigor. Lower classes draw fewer, lighter requirements through this matrix.
- **The five underlying factors** — usage, system criticality, human dependence, developmental/operational complexity, and Agency investment — invoked both to define the classes and to resolve disputes about them.
- **Dual TA chains (Engineering and SMA)** — two Technical Authority chains, one in Engineering and one in Safety and Mission Assurance, that resolve classification questions. The directive names the **NASA Chief Engineer** as the final Technical Authority for classification disputes over the definitions in this NPR.
- **External standards referenced by the class definitions** — the definitions lean on other documents, including NASA-STD-3001 Volumes 1 and 2 (crew health; human factors, habitability, and environmental health), the FAA Human Factors Design Standard (HFDS), DO-178C (*Software Considerations in Airborne Systems and Equipment Certification*, whose Class D and Class E levels are cited by analogy), NPR 7120.8 (the large-scale technology-development threshold), and NPR 8705.4 (NASA payload risk classification, e.g., Class D payloads).

## Key Concepts

**One class per system, round up on ambiguity.** Software for a given system or subsystem is expected to fall uniquely into a single class. When more than one class seems to fit, the directive resolves the tie by assigning the higher class. Discrepancies inside the A-through-E range are settled using the definitions plus the five factors, and ultimately by the Chief Engineer.

**Mixed-class projects are normal.** It is routine for one project to contain separate systems and subsystems carrying different classes. The classification is applied at the system/subsystem grain, not project-wide.

**Class A — Human-Rated Space Software Systems.** Ground and flight software developed or operated by or for NASA that is needed for a primary human-spaceflight mission objective and interacts directly with human-spaceflight systems. It is bounded to software that operates the vehicle or space asset (spacesuit, rover, outpost) including commanding it, sustains a safe and habitable crew environment, achieves the primary mission objectives, or directly prepares resources (data, fuel, power) consumed by those functions. It spans launch, on-orbit, in-space, surface operations, and entry/descent/landing (EDL). Habitability standards it relies on are documented in NASA-STD-3001 and the FAA HFDS. Class A *excludes* software that merely rides along but is superfluous to the mission (an astronaut's personal music player), software that only supports aeronautics/research/science with no spaceflight application, and development-environment test rigs (simulators, emulators, facilities) used to test Class A systems.

**Class B — Non-Human Space-Rated Systems or Large-Scale Aeronautics Vehicles.** Flight and ground software that must perform reliably to accomplish primary objectives or major functions in non-human space-rated systems — operating the asset (orbiter, lander, probe, flyby spacecraft, rover, launch vehicle, primary instrument), achieving primary objectives, or preparing resources consumed by those functions. It also covers large-scale NASA-unique aeronautic vehicle software that is integral to vehicle control, or that monitors and controls the cabin environment or emergency systems. "Large-scale" here is pinned to a life-cycle cost above $250M for a fully integrated technology-development system, per NPR 7120.8. Test/experimental/demonstration vehicles qualify if their software meets the Class B definition or is scheduled for known use as part of a Class A or B system. It *excludes* software that only supports non-primary instruments (e.g., low-cost university-supplied instruments), development test rigs for Class B systems, and software for NASA Class D payloads under NPR 8705.4.

**Class C — Mission-Support, Smaller Aeronautics, or Major Facility Software.** A broad middle tier. For space systems it covers flight/ground software for science return from a single non-primary instrument, mission-data analysis and processing, software whose defect could harm secondary objectives or cause operational problems, software used to test space assets or to verify space-asset requirements by analysis, and space-flight operations software not already Class A or B. For aeronautics it covers non-large-scale vehicle control, cabin, and emergency software, plus systems on any vehicle (large-scale included) that hold the official record copy of flight or test data. It also captures major engineering/research facility control and monitoring, sounding rockets and their payloads, and all software on NASA Class D payloads. Example venues include Distributed Active Archive Centers, CubeSat and SmallSat mission software, wind-tunnel and rocket-engine-test-stand data acquisition, and high-fidelity motion-based simulators. It *excludes* research/test/evaluation systems that sit in a major facility or aboard a vehicle but are not part of it and do not affect its operation.

**Class D — Basic Science/Engineering Design and Research-and-Technology Software.** Mostly ground software: secondary science-data analysis, engineering-development tools, informal software-test tools, mission-planning/formulation tools, software operating a non-major RDT&E laboratory, and decision support for non-mission-critical situations. Its airborne-vehicle tier maps to the lowest DO-178C consequence levels (the directive cites DO-178C "Class D" for a minor failure condition and "Class E" for no operational effect), plus ground research tools tied to airborne systems. Examples include CAD/CAM and thermal/structural tools, problem-reporting and requirements-management databases, and flight software for physically constrained UAVs (tethered, caged, or indoor-lab). It *excludes* anything able to affect primary or secondary mission objectives, or to lose space-system data; anything that operates a major facility or an airborne vehicle; and any flight software that already meets the flight portions of Classes A, B, or C.

**Class E — Design Concept, Research, Technology, and General-Purpose Software.** Software built to investigate a design concept or hypothesis that is not relied on for decisions about an operational or planned Class A/B/C system, software doing minor analysis of science or experimental data, and general-purpose tools whose worst-case defect reaches only one user or a small group. It runs in a general-purpose computing or bench-top environment and does not support ground tests, flight tests, or operations. A hard floor applies: **Class E can never be safety-critical** — safety-critical software must be Class D or higher. Examples include parametric performance models, line-of-code counters, file-format converters, word processors, and spreadsheets, and even minimal-cost prototypes of flight/ground systems "in the spirit of exploring a design concept." Its exclusion list is the longest: it rules out flight systems, software supporting or qualified for an operational system, anything affecting mission objectives, anything that can corrupt engineering/scientific artifacts, software used in technical decisions about operational systems, anything impacting operational vehicles, and anything safety-critical.

**Class F — General-Purpose Computing, Business, and IT Software.** The separate Business/IT track. It is general-purpose computing and business/IT software supporting the Agency, multiple or single Centers and programs/projects, or locally deployed general-purpose infrastructure within the NASA Enterprise Architecture — voice, networks, video, data centers, cloud computing, information management, business/IT application services (Finance, Logistics, Human Capital, Procurement), messaging and collaboration, and the public Web. A defect here typically affects the productivity of many users across one or several locations and may affect cost, schedule, or technical mission objectives. Examples include Agency-wide enterprise applications (WebTADS, SAP, CONCURGov, ePayroll, Business Warehouse), Center-specific and Web/mobile applications, the NASA wide-area network, and the NASA Web portal.

## Mental Models

**A consequence ladder, not a complexity ladder.** Walking A→B→C→D→E, the question that moves you down a rung is always "what is the worst that a defect does, and who depends on it?" Human life and primary mission (A), reliable non-human mission/large vehicle (B), secondary objectives and mission support (C), development and basic research (D), throwaway exploration and desktop tools (E). Sophistication of the code is irrelevant; a simple ground command that aborts a crewed launch is Class A, while an elaborate CAD package is Class D.

**The "round-up" safety bias.** Ambiguity never resolves downward. When two classes both seem plausible, the directive forces the higher (more rigorous) one. This is a deliberate fail-safe: misclassifying upward costs effort; misclassifying downward costs assurance on something that needed it.

**The safety-critical floor.** Class E has a hard tripwire — the moment software is safety-critical, it cannot be E and must be reclassified to D or higher. Safety-criticality is a one-way ratchet against the bottom of the ladder.

**Use-case promotion over the life cycle.** A class is not a permanent label stamped at birth. Prototype software built cheaply as Class E to demonstrate a concept must be upgraded to its proper class the moment a program commits to flying it or using it operationally (or in an in-flight technology test). Classification tracks intended use, and intended use can change.

**Two-axis location: venue plus consequence.** Where software runs (space, air, ground, facility, desktop) narrows the relevant definition branch, but consequence and dependency pick the class within that branch. "Flies in space" is not Class A; "flies in space and the crew depends on it for a primary function" is.

**Dual authority with a single tiebreaker.** Engineering and SMA Technical Authority chains both have standing in classification, reflecting that class drives both engineering rigor and safety assurance. But disputes over the NPR's own definitions converge on one person: the NASA Chief Engineer.

## Anti-patterns

The directive does not enumerate named anti-patterns in this slice. It does, however, repeatedly fence off specific *misclassifications* through its exclusion lists — the recurring error it guards against is treating co-location or coincidental presence as evidence of class (software that merely rides aboard a vehicle, sits in a facility, or happens to fly in space), when the governing test is whether the software is part of, and consequential to, the mission, vehicle, or facility function.

## Key Takeaways

- Six classes govern applicability: A–E for engineering-related software, F for Business/IT infrastructure software.
- Classification is decided by five factors — usage, system criticality, human dependence, developmental/operational complexity, and Agency investment — not by code size, language, or runtime venue.
- Class determines requirement set and rigor via the Requirements Mapping Matrix; lower classes get fewer, lighter requirements.
- One class per system/subsystem; on ambiguity, assign the higher class. Mixed-class projects are expected.
- Each class definition carries explicit exclusions, mostly to stop co-location from being mistaken for consequence.
- Class E can never be safety-critical; safety-critical software is Class D or higher.
- Classification follows intended use and can be promoted over the life cycle (e.g., a Class E prototype committed to operational use must be upgraded).
- Engineering and SMA share dual Technical Authority over classification; the NASA Chief Engineer is the final arbiter of definition disputes.

## Connects To

- **The Requirements Mapping Matrix and the directive's requirement set** — classification is the upstream decision that feeds the matrix; everything about how many NPR 7150.2D requirements bind a project flows from the class assigned here.
- **NASA-STD-3001 (Vols 1 and 2) and FAA HFDS** — the human-system/habitability standards the Class A definition relies on for "safe, habitable environment."
- **DO-178C** — the airborne software assurance standard whose lower levels (its Class D and Class E) the Class D definition cites by analogy for airborne-vehicle software.
- **NPR 7120.8** — supplies the large-scale ($250M life-cycle-cost) threshold that distinguishes large-scale aeronautics (Class B) from non-large-scale (Class C).
- **NPR 8705.4** — NASA payload risk classification; its Class D payloads are routed into software Class C and excluded from software Class B.
- **Technical Authority structure (Engineering and SMA chains; NASA Chief Engineer)** — the governance backbone that adjudicates classification, linking this appendix to NASA's broader institutional authority model.
