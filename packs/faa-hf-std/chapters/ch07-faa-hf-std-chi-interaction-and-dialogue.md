# Chapter — Computer-Human Interface: Interaction, Windows, and Dialogue

> Source: FAA *Human Factors Design Standard* (HF-STD-001B), Sections 5.6.7 through 5.6.15 — the computer-human interface material spanning interaction methods, menus, function keys, general interactive techniques (direct manipulation, command and query languages), user-initiated interrupts, file management, selection methods, transaction options, controls (icons, push/radio/check buttons, list boxes, sliders, cursors), windows and message windows/dialog boxes, and system operations (log on/off, response time, prompting, feedback, error management). Every requirement, definition, and discussion paraphrased here is drawn from those sections. The standard attributes most of its rules to underlying sources — MIL-HDBK-761A, DOD-HFDG-ATCCS V2.0, MIL-STD-1801, DON UISNCCS, Ameritech Services Inc., Apple Computer Inc., Microsoft Corp., DSTL-95-033, NUREG-0700, ESD-TR-86-278, National Air Traffic Services, DISA, and others; those citations are the standard's own.

## Core Idea

This run of HF-STD-001B is the requirement-level rulebook for how an operator and a computer talk to each other through a graphical interface. Where the automation chapter governed *what the machine decides*, this material governs *how the person drives it* — the interaction style, the widgets, the windows, and the conversation of prompts, feedback, and error recovery that surrounds every transaction. It is dense and concrete: hundreds of individually-cited `shall` and `should` statements, organized into nested topic clusters, each carrying its own definition and often a worked example.

The organizing logic runs top-down through the interaction stack. First the standard makes the designer *choose an interaction type* (5.6.7.1) — question-answer, form-filling, menu, function key, command language, query language, constrained natural language, or direct manipulation — matched to the task, the system, and the user's training. Then it details each style's mechanics: menus get the largest treatment by far (seventeen sub-sections covering titles, options, wording, organization, and a dozen menu variants), followed by function keys, direct manipulation, and command/query languages. After interaction methods come the supporting layers: how to interrupt and undo (5.6.9), manage files (5.6.10), select data (5.6.11), stack and option transactions (5.6.12), the control widgets themselves (5.6.13), the window system that contains them (5.6.14), and finally system-level operation — log on/off, response time, prompting, feedback, and error management (5.6.15).

Two motifs recur in nearly every cluster. *Consistency* — the same menu, option, function key, command word, button order, title format, or prompt location should behave identically within and across related applications. And *feedback with recoverability* — every user action should draw a system response, destructive actions demand confirmation (and not via the Enter key), and a nondestructive way back (Undo, Cancel, Back, Restart) should always exist.

This chapter is grounded entirely in the FAA standard. It is human factors guidance written for air-traffic and aviation-domain interactive systems — the design-rule layer, distinct from but complementary to the NASA human-systems-integration body of knowledge. NASA HSI frames human-systems integration as a lifecycle discipline woven into the systems-engineering engine; HF-STD-001B supplies the concrete, citable interface requirements an engineer applies once a screen is actually being built. The two are meant to be read together, and the caveat from elsewhere in this pack holds: HSI extends well beyond the NASA treatment, and this domain-specific FAA material is one of those broader sources.

## Frameworks Introduced (exact source names, when/how)

- **Interaction-type selection (5.6.7.1.1).** The decision framework that opens the whole section: eight named interaction types, each with the task and user conditions under which it fits. Question-answer suits routine, constrained data entry by untrained users; form-filling suits flexible entry by moderately trained users; menu selection suits choices from constrained sets with little training; function keys suit a limited set of fast critical entries; command language suits wide-ranging entries by highly trained frequent users; query language suits unpredictable retrieval by trained users; constrained natural language suits wide or poorly-defined tasks with moderate training; and direct manipulation suits acting on concrete on-screen objects and casual or untrained users. This is the first thing to apply — pick the style before designing the screen.

- **Menu-variant taxonomy (5.6.7.5).** A worked classification of menu forms, each defined and rule-governed: menu bars, the system menu and system menu bar, pull-down, hierarchical, cascading, scrolling (including spin boxes and list boxes), pop-up, tear-off, toggled, and graphic menus (palettes). Definitions are explicit — e.g. a menu is a list of options, an option is one selectable item, selection is the act of choosing one. Used to pick the right menu mechanism for the option count and screen budget.

- **Mnemonics vs. keyboard accelerators (5.6.7.6.2).** A two-part keyboard-shortcut model. A mnemonic is a single underlined letter typed *while the menu is displayed* (preferably the option's first letter, unique within the menu). An accelerator is a key or key combination (e.g. Ctrl+S) that fires the option *without* displaying the menu. The framework includes the trade-off rationale — letters beat numbers for memorability and there are more of them, but numbers clarify sequence and count.

- **Direct manipulation (5.6.8.1).** Defined as a GUI dialogue in which the user controls the computer by acting directly on display objects (icon, menu option, symbol, button, dialog box). Recommended for casual users and as a simpler alternative to learning a command language when response time is fast; pairs windows (to contain the data file) with menus (for objects not easily iconified).

- **Command language and query language (5.6.8.2, 5.6.8.3).** Two text-driven dialogue frameworks. A command language is a limited programming language for executing command series; the rules govern consistent syntax, distinctive and familiar wording, minimized punctuation, abbreviation, command stacking, editing-before-execution, and confirmation of destructive commands. A query language is for specifying, locating, and retrieving data from a database; its rules add task-oriented "say what, not how" querying, logical combinations, iterative refinement, large-retrieval confirmation, interruptibility, and a full catalog of search, control, and formulation operations (5.6.8.3.3).

- **User-initiated interrupt set (5.6.9.1.2).** A defined family of distinctly-named interrupt controls — Cancel, Escape, Back, Restart, Abort, Stop, Pause-Continue, Suspend — each with its own control option. Anchored by the rule that interrupts must not alter stored or entered data (except Cancel), plus Back/Go-back, Undo/Redo, End/Exit/Stop, Pause/Continue, Restart/Revert, Review, and Suspend, each defined by its effect on the transaction.

- **Selection model: anchor point and active end (5.6.11.2).** A geometric model of text/graphic selection. The selection runs between a fixed anchor point and a movable active end; extending moves the active end away from the anchor, shortening moves it back, and when they meet the selection collapses to an insertion point. Includes the named selection techniques — clicking, dragging, command-click for discontinuous selection, outline selection, and margin selection.

- **Two-action activation (5.6.7.6.1.8).** The pointing-selection model: activation is *designation* (position the cursor on the option) followed by a *separate explicit* activation (click). The deliberate point-then-click is preferred over press-drag-release because it is less error-prone (5.6.7.5.14.8).

- **Window modality and hierarchy (5.6.14).** The foundational window vocabulary. Windows are modal (the user must deal with them before any other window) or modeless. Windows are primary (top-level, independent) or secondary (child windows opened from another). Input focus is the notion that only one window — and usually one object within it — accepts pointer/keyboard input at a time, with focus assignable explicitly (move-and-click) or implicitly (move only).

- **Message-window family (5.6.14.4).** A typed set of special-purpose secondary windows, each with prescribed symbol and button set: request (`?`), information (`i`, modal, OK/Help), confirmation (`?`, Yes/No[/Cancel]/Help), warning (`!`, plus an audible signal, suspends processing), working (a watch/hourglass symbol, ≥2-second processes, non-interrupting), error dialog (error symbol, modal, OK/Cancel/Help), and dialog boxes / alert boxes. This is the standard catalog for "the system needs to say something."

- **System-response-time and feedback model (5.6.15.5, 5.6.15.7).** Quantified timing rules tying perceived responsiveness to design behavior: response variability bounded (≤5% under 2 s, ≤10% for 2–5 s, ≤15% beyond 5 s); a "working" indication required past 2 seconds; periodic feedback past 2 seconds; user informed when delay will exceed 15 seconds; a countdown past 60 seconds; and a clear positive (e.g. audible) completion signal when response exceeds 15 seconds.

- **Error-management framework (5.6.15.10).** A structured approach to user errors: detect fast, respond to *every* entry (correct or not), correct in place, go back easily, undo/redo, and require explicit confirmation (not the Enter key) before destructive actions. Error messages get their own rules — brief, specific, task-oriented, stating the error, the field, and the corrective action; multilevel on request; non-disruptive (not shown mid-entry); and delivered within 2 to 4 seconds of the entry.

## Key Concepts

- **Match the interaction type to the user, not the designer (5.6.7.1.1).** The chapter's first move forces an explicit fit between interaction style and the triad of task requirements, system characteristics, and user abilities/training. Choosing command language for untrained casual users, or menus for high-volume expert entry, is a category error the framework is built to prevent.

- **Consistency is the dominant `shall`.** It recurs in nearly every cluster: question-answer format (5.6.7.3.1), form structure (5.6.7.4.1), a single menu style across an application (5.6.7.5.1.4), identical wording/organization for repeated menus and options (5.6.7.5.1.5), command syntax (5.6.8.2.3), function-key assignment within and across applications (5.6.7.7.3–7.7.4), window organization and elements (5.6.14.1.10–1.12), push-button order (5.6.14.2.8.4), and prompt location (5.6.15.6.3). The implicit model: consistency is what lets a learned action transfer, so the standard treats inconsistency as a defect.

- **Breadth-over-depth in menus (5.6.7.5.11.5, with 5.6.7.5.1.8).** Hierarchical menus should reach any option in no more than four selections, and broad-shallow structures are preferred over narrow-deep ones. The nuance: menu length should also track system timing — when response time is long, make menus broad and shallow; when the *draw* rate is slow, make them narrower and deeper. Per-menu option count should sit between 3 and 10.

- **Mutually-exclusive choice scales with option count (5.6.7.5.1.2).** A clean sizing rule for single-choice controls: radio buttons up to 6 options, a menu up to 10, a scrolling menu beyond 10. Radio-button sets themselves should hold two to seven items (5.6.13.6.14), spilling to a scrollable or drop-down list at nine or more.

- **Distinguish, disable, but still reveal (5.6.7.5.3.6–5.3.8).** Temporarily unavailable options stay displayed but visibly distinct (e.g. grayed out); a menu whose options are all unavailable is itself disabled; yet the user can still pull a disabled menu down to inspect it. The interface hides nothing — it dims.

- **Every action gets a response (5.6.15.1.5, 5.6.7.5.3.9).** A blanket `shall`: every user action produces a system response, and no system function activates without an indication to the user. When a selection yields no immediately observable result, the software must acknowledge it some other way — a watch/hourglass or an elapsed/remaining-time message.

- **Destructive actions are fenced (5.6.15.10.1.11–10.1.12, 5.6.8.2.25).** Control entries that change or could lose stored data require explicit notification of the consequence and a confirming action — and that confirmation `shall not` be the Enter key. Destructive menu commands (Delete, Exit) go at the bottom of frequency-ordered menus, and opposing actions (Delete next to Save) must not be adjacent. A potentially destructive push button cannot be the default button.

- **Recoverability is layered (5.6.9, 5.6.15.10.1.14–10.1.15).** Nondestructive Back, Cancel/Undo (which restores prior state), Redo (or a second Undo to reverse the undo), Restart/Revert, Review, and Suspend each give a different escape route. Undo immediately reverses the last control action; a second Undo reinstates it.

- **The window contains everything else.** Windows have a defined component vocabulary — title bar (with window-management control on the left; Minimize/Maximize-Restore/Close on the right), border, working/client area, scroll-bars, message bar, status bar, control bar, and push buttons. The active window is the one with input focus; only one is active at a time, it is never obscured, and it sits in front of overlapping windows. Closing a primary window removes its secondaries; closing a secondary leaves the parent intact.

- **Scroll-bar as proportional map (5.6.14.2.4.11–2.4.14).** The scroll-bar's length represents the whole entity and the scroll box's *size* indicates proportionally how much of the document is currently shown — a long document yields a small box. The box's position shows where in the file the view sits. Vertical scrolling is preferred to horizontal where there's a choice.

- **Modal only when it must be (5.6.14.3.1.18, 5.6.14.4.7.2).** Modal secondary windows and modal dialog boxes are reserved for serious situations needing an explicit response before continuing; modeless windows handle anything that does not demand immediate attention. Information, confirmation, warning, and error windows are explicitly modal because they require acknowledgement.

- **Response time is a designed property (5.6.15.5).** Response must suit the transaction and the task's time constraints, and — notably — its *variability* must be minimized within bounded percentages by processing band. Slow processing triggers a graded feedback ladder (working indication at 2 s, inform-of-delay at 15 s, countdown at 60 s), and keyboard lockout during processing must be indicated, minimized, and overridable without losing completed work.

- **Login security in the interface (5.6.15.2).** Log-on rules carry security into HF: show the user name but never the password; on a log-on error, give guidance to correct it but nothing that would help an intruder; confirm log-off, warn before automatic inactivity log-off, and auto-save on automatic log-off.

## Mental Models

- **The interaction stack.** Read the section ordering as a stack the designer descends: choose interaction *type* → design that type's *mechanics* (menus, keys, language) → wire the *control widgets* (buttons, lists, cursors) → place them in the *window system* → wrap it in *system-level* conversation (prompts, feedback, errors, log on/off). Each layer assumes the one above it is settled.

- **Designation then activation.** The pointing model treats every selection as a two-beat action — aim, then commit — and the standard prefers this deliberate two-beat over a continuous press-drag-release precisely because separating intent from commitment reduces slips. The same logic underlies requiring an explicit Enter/execute after composing a command (5.6.8.2.23) and an explicit Save/Apply/OK to commit data-entry (5.6.14.3.3.6).

- **Anchor-and-active-end selection.** Selection is not a static highlight but a span anchored at one fixed point with a movable active end; extending, shortening, and collapsing to an insertion point are all just moving the active end. This model unifies mouse drag, Shift+navigation-key keyboard selection, and the insertion point into one mechanism.

- **Input focus as a single spotlight.** Across any number of open windows, exactly one window — and usually one object inside it — holds focus and receives input. Focus is shown by a location cursor or highlight, it survives a window losing and regaining activity (returning to the last-focused object), and it is the user's to assign, not the application's to grab (a closed focused window does not auto-reassign focus unless an emergency requires it).

- **Modality as an interruption budget.** Modal windows spend the user's attention forcibly; modeless ones let it flow. The framework's posture is to spend modality sparingly — only for serious problems and required inputs — because every modal window is a forced stop in the user's task.

- **Feedback graded to delay.** The user's tolerance for silence is treated as a decaying function of time: acknowledge instantly, show "working" by 2 seconds, warn of a long wait by 15 seconds, count down past 60 seconds, and signal completion positively when the wait was long. The model assumes that *uncertainty*, not delay itself, is what frustrates the operator.

- **Trained-vs-untrained as a design axis.** The same axis that picks the interaction type runs through the whole section: novices get prompts, lists, fewer/simpler commands, and step-by-step guidance, while experts get shortcuts, accelerators, command stacking, macros, and the ability to bypass guidance (5.6.8.3.5, 5.6.7.6.1.9, 5.6.7.5.11.14). A good interface serves both ends of the axis at once rather than choosing one.

## Anti-patterns

The standard names several interface failure modes directly or fences them with `shall not` rules:

- **Dead-end transactions (5.6.12.1.17).** A transaction should never leave the user with no available next option; named explicitly as something to avoid, with the remedy being to always offer next steps or alternatives at completion (5.6.12.1.18).
- **Getting lost in deep menus (5.6.7.5, 5.6.7.5.12.4).** The standard warns that complex menu sequences let users become lost in the structure, and that cascading menus force users to remember where buried options live — hence the four-level cap, one-level cascade limit, current-position indicators, and easy return-to-top/return-to-higher-level rules.
- **Spelling-induced command errors (5.6.8.2.14).** Commands chosen so that a likely typo produces a *different valid* command — the named example is DEL/SEL colliding because D and S are adjacent on QWERTY. The rule: pick command words so probable misspellings do not resolve to valid commands.
- **Adjacent opposing actions (5.6.7.5.6.9).** Placing options for opposing actions next to each other (the named example: Delete beside Save) invites destructive slips and is prohibited.
- **Abstract and humorous icons (5.6.13.3.11, 5.6.13.3.8).** Abstract icons are flagged as hard to learn and remember and should be avoided; humorous representations should be avoided in icons.
- **Enter key for destructive confirmation (5.6.15.10.1.12).** Using the same Enter key the user reflexively presses to also confirm destructive actions is prohibited — the confirmation must be a deliberate, different action.
- **Flash rates that induce seizures (5.6.13.10.2.4).** Display refresh or cursor flash within the 15–20 Hz band can induce seizures in sensitive users and is prohibited; rates must sit well above or below that band (cursor flash is specified at 2–5 Hz, preferably 2–3 Hz at 50% duty cycle).
- **Silent slow processing.** Implied throughout 5.6.15.5/5.6.15.7: an operation that takes seconds with no working indication, no periodic feedback, and no completion signal violates the response-time and feedback rules.

## Key Takeaways

- Begin at the top of the stack: explicitly select the interaction type (5.6.7.1.1) against task, system, and user-training before designing any screen. The eight named types are a decision tool, not a menu of equally-valid styles.
- Treat consistency as a hard requirement, not a nicety — identical wording, organization, syntax, key assignment, button order, window layout, and prompt placement within and across related applications. Inconsistency is a defect.
- Size choice controls to option count (radio ≤6, menu ≤10, scrolling beyond), keep menus 3–10 options, and prefer broad-shallow hierarchies capped at four levels; tune menu shape to system response and draw rates.
- Engineer recoverability and confirmation first: every action gets a response, destructive actions get explicit non-Enter confirmation, opposing/destructive options are separated, and Back/Cancel/Undo/Redo/Restart/Suspend give layered escape routes.
- Use the message-window family by the book — each type carries a prescribed symbol and button set, modality is reserved for serious or required-input cases, and warnings add an audible signal.
- Make slow systems honest: bound response-time variability, show a working indication by 2 seconds, warn of long delays by 15 seconds, count down past 60, signal completion positively, and never lock the keyboard silently or unrecoverably.
- Serve novices and experts on the same axis: prompts, lists, and guidance for the untrained; accelerators, stacking, macros, and bypasses for the trained — in one interface.
- Honor the perceptual and safety floors: icons concrete and standard (not abstract or humorous), flash rates clear of the 15–20 Hz seizure band, passwords never shown, and login errors guided without leaking intrusion-useful detail.
