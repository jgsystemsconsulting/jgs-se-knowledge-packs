# Chapter 6: Define the Software Requirements and Allocate System Requirements to Subsystems

## Core Idea
Once a system's behavior is captured as relationships between **monitored** and **controlled** variables, software requirements can be written as a disciplined *extension* of the system requirements rather than as a separate, loosely-coupled document. The handbook builds this bridge on the **four-variable model**: software cannot touch the real-world monitored and controlled quantities directly, so the system hands it input variables to read and output variables to write, and the software's job is to reconstruct images of the monitored variables, apply the same input/output relationship the system requirements already specified, and drive the outputs. The same monitored/controlled-variable discipline then scales upward: large systems are carved into independently developable subsystems by allocating system functions, exposing internal variables as new monitored/controlled variables at the subsystem boundary, and duplicating high-level requirements to preserve traceability. This chapter carries a deliberate **embedded / real-time slant** — every example turns on accuracy, latency, bit-level data formats, and end-to-end timing budgets — and it complements, rather than restates, the earlier requirements-writing guidance.

## Frameworks Introduced
- **The Four-Variable Model** (attributed to Parnas and Madey, from the SCR methodology). The handbook uses it as the formal backbone for relating system to software requirements. It names four sets of quantities and the relations among them:
  - **MON** — the environmental quantities the system monitors; **CON** — the environmental quantities the system controls.
  - **NAT** — the environmental assumptions (relationships the environment maintains that the system depends on); **REQ** — the relationship the system must maintain between MON and CON.
  - **INPUT / OUTPUT** — the variables the software can actually read and write, which the system translates to and from MON and CON.
  - **IN / OUT** relations — how each input relates to monitored variables (and each controlled variable to outputs), each carrying an ideal-value function, a latency, and an accuracy.
  - When to use: as the structuring frame whenever you need to transition cleanly from a system spec to a software spec for a sensor-driven or actuator-driven function.
- **Extended Software Requirements (IN', REQ', OUT')**. Because the raw software relation (SOFT) hides the mapping from system to software, the handbook splits it into three parts (citing reference [44]):
  - **IN'** — the inverse of IN: how to recreate the monitored-variable images (**MON'**) in software from the inputs.
  - **OUT'** — the inverse of OUT: how the outputs change as the controlled-variable images (**CON'**) change in software.
  - **REQ'** — the relationship the software maintains between MON' and CON'; its ideal-value function is identical to the system's REQ.
  - When to use: to make the system-to-software mapping direct and to separate concerns — REQ' tracks changes in *system requirements*, while IN' and OUT' track changes in *hardware* (think of them as driver specifications).
- **High-Level vs. Low-Level Software Requirements** (from RTCA DO-178B, with DO-248B and DO-178B cited as the governing references). The handbook maps its artifacts onto DO-178B's two-tier structure: low-level requirements (IN', REQ', OUT', and the detailed system functions) are those from which source code can be implemented directly; high-level requirements are the remaining system-level items (dependency diagrams, internal-variable definitions, higher-level functions).
  - When to use: when a project must show compliance under DO-178B objectives — high-level requirements complying with system requirements, low-level complying with high-level.
- **Subsystem Allocation Procedure** (Section 2.10, recommended practices 2.10.1–2.10.10). A ten-step method for splitting a system into independently developable subsystems while preserving the variable-based discipline and traceability.
  - When to use: on large systems where functions are spun off to subcontractors and each subsystem needs to stand as a complete specification in its own right.

## Key Concepts
- **MON' and CON' are not MON and CON.** The images recreated in software always lag and deviate slightly from the true environmental quantities — hardware and software introduce both error and latency. The handbook uses a prime notation (e.g. *airspeed'*) to keep the software image distinct from the real quantity, and insists these differences be carried through the timing and tolerance analysis.
- **Inputs carry three things: definition, accuracy, latency.** For every input the software reads, the spec must give enough for a developer to access and interpret it (description, data format, value range, memory location, access protocol), plus an accuracy bound (how far it may deviate from ideal) and a latency bound (how far it may lag the true monitored value).
- **Bit-level concreteness is the point.** The handbook's worked examples are deliberately low-level: a monitored altitude that is a higher-abstraction integer becomes an ARINC 429 bus word with a 2's-complement encoding across specific bit positions; the Isolette current temperature becomes a 16-bit unsigned integer (temperature × 256, eight bits of fraction) at two contiguous memory words. This is the embedded/real-time register in which software requirements actually live.
- **Status attributes ride alongside values.** Every monitored variable carries a health/status attribute showing how well it can be sensed, initialized to *invalid* until first successfully read. The IN' relation must specify how to set the status exactly as carefully as it specifies how to set the value (e.g. a separate "status in" input variable plus saturation-range checks driving Valid/Invalid).
- **IN' can be one-to-one or many-to-one.** Recreating a monitored-variable image may be a simple mapping from a single input or a complex function of several inputs.
- **Outputs mirror inputs, but accuracy/latency attach to the controlled variable.** For outputs, latency is the lag from setting the output until the controlled variable changes, and accuracy is how far the controlled variable may diverge from its ideal — both specified against the *controlled* variable, not the output variable. Discrete outputs take N/A for accuracy.
- **REQ' comes for free.** With INPUT/OUTPUT variables and IN'/OUT' defined, the spec is nearly complete: the only remaining behavioral piece, REQ', reuses the identical ideal-value function from the detailed system requirements.
- **End-to-end timing must be confirmed, not assumed.** The last software step is checking that each controlled variable's overall latency/accuracy budget survives the input latency, output latency, and software computation time. The handbook's Isolette worked figure: a multi-second controlled-variable latency budget, minus sub-millisecond sensing and actuation lags, leaves the computation a defined slice of seconds to finish in.
- **Subsystem boundaries expose internal variables.** When a function is allocated to a subsystem, variables that were internal to the parent become genuine monitored/controlled variables at the subsystem interface — they must be defined, made consistent with the parent's internal-variable definitions, and given environmental assumptions.
- **Duplication is deliberate, for traceability.** A copy of an allocated function and its high-level requirements stays in the parent spec so each subsystem's topmost requirement traces directly to an identical parent requirement.
- **A subsystem spec must become complete on its own.** Beyond copied functions it needs a system overview, monitored/controlled variables, operational concepts (e.g. use cases), and environmental assumptions, after which the functional decomposition continues down to detailed behavior and performance requirements.
- **Latency budgets compose across subsystems.** The summed latencies of the controlled variables along a chain of subsystems must stay under the end-to-end latency specified for the corresponding controlled variable in the parent; tolerances must compose consistently the same way.

## Mental Models
- **Software requirements = system requirements + four-variable wrapper.** Don't think of the software spec as a fresh document; think of it as the system spec plus the INPUT/OUTPUT variable definitions and the IN/OUT/IN'/OUT' relations bolted on. The behavior in the middle (REQ') is unchanged.
- **IN' and OUT' are the driver layer; REQ' is the application layer.** The split isolates the part that changes when hardware changes (IN'/OUT') from the part that changes when system behavior changes (REQ'). That separation of concerns insulates the behavioral software from platform churn.
- **The prime mark is a humility reminder.** Every *variable'* in software is a delayed, slightly-wrong copy of reality. Treating MON'/CON' as if they were the real thing is how timing and tolerance assumptions silently break.
- **Saturation choices are requirements, not coding details.** Clamping a recreated value to a physically-assumed range (because an environmental assumption says it stays there) is a design decision that affects externally visible behavior — so it is a *requirement* the safety process must see, not a private implementation convenience.
- **Allocation is fractal.** A subsystem, once specified, is just a smaller system: you re-run the same steps (overview, variables, operational concepts, environmental assumptions, decomposition, safety analysis, software requirements) inside it.

## Anti-patterns
- **Burying derived design decisions.** The handbook names *derived requirements* (per DO-178B / DO-248B) — requirements with no direct trace to a higher-level requirement, such as the choice to saturate a recreated temperature at its assumed range. Because alternative choices (e.g. a wider clamp range) would change externally visible behavior and could trade reduced nuisance-failure risk for the possibility of operating at a dangerous value, such decisions must be *flagged* as derived software requirements and routed to the system safety assessment — not left implicit in the code.
- **Including parent variables the subsystem doesn't use.** When identifying a subsystem's shared monitored/controlled variables, the handbook explicitly warns against carrying over monitored/controlled variables from the system spec that the subsystem does not actually use (the FGS example drops fields and controlled variables the FCS had but the FGS does not touch).

## Key Takeaways
1. The **four-variable model** (MON, CON, NAT, REQ, plus INPUT/OUTPUT and IN/OUT) is the formal bridge from system to software requirements; the handbook attributes it to Parnas and Madey via SCR.
2. Software requirements are an **extension** of system requirements: define the INPUT/OUTPUT variables and the **IN'/REQ'/OUT'** relations, and REQ' reuses the system's existing ideal-value function unchanged.
3. **IN'/OUT' isolate hardware concerns; REQ' isolates behavioral concerns** — a separation that protects behavioral software from platform changes.
4. Every input/output needs a **definition, an accuracy bound, and a latency bound**, plus a **status attribute** initialized to invalid; examples are deliberately bit- and register-level.
5. Always **confirm the end-to-end latency and accuracy budget** survives sensing lag + actuation lag + computation time for each controlled variable.
6. Map artifacts onto **DO-178B's high-level / low-level requirements** so compliance can be shown; flag **derived requirements** (design choices affecting visible behavior or safety) to the safety assessment.
7. Allocate large systems to subsystems by **duplicating high-level requirements** for traceability, **exposing internal variables** as subsystem monitored/controlled variables, and making each subsystem spec **complete in its own right**.
8. Ensure **composed subsystem latencies and tolerances** stay within the parent system's end-to-end budgets.
