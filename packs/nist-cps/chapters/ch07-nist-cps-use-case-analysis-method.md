# Chapter 7: Use Case Analysis Method and Worked Examples (Vol 2, Use Case Sub-group)

## Core Idea
The set of possible cyber-physical system (CPS) uses is effectively unbounded, so you cannot enumerate every use case. Instead, the Use Case Sub-group built a repeatable, two-stage analysis method: collect broad **CPS Examples** that span several systems and application domains, then deconstruct each into single-actor/single-system **specific use cases** whose steps map to a library of **primitive (simple) requirements**. Evaluating each use case consistently against a fixed set of **architectural characteristics** lets you cluster them, spot coverage gaps (e.g., no use case with tight timing), and abstract the results into requirements levied on the reference architecture and the other sub-groups. Use cases are the link between each user's goals and the system properties the CPS Framework must support.

## Frameworks Introduced
- **Two-stage use case process (CPS Examples → specific use cases)** — Section 5.2. Stage one collects high-level CPS Examples that may cross application-category boundaries and contain multiple systems, actors, and interactions; this surfaces *which* requirement areas matter and *who* the actors and systems are. Stage two breaks each Example into specific use cases (one actor, one system) and attaches primitive requirements to individual steps to get the precise, testable requirements needed to validate the architecture. Use when you need both a wide-coverage survey and the fine-grained requirements that survey alone cannot give.
- **CPS Example template** — Table 6, Section 5.2.1. A use-case *summary* form capturing Name, Description, Notes, Goals, Source Organization, Actors, and Systems. Use to record a multi-system scenario before decomposing it.
- **Black-box specific use case template** — Table 8, Section 5.2.3. Adds Pre-Conditions, Steps, and Post-Conditions to the summary fields, following traditional use-case design centered on actor, system, and the steps between them. Use to describe one specific actor-system interaction in enough detail to derive step-level requirements.
- **Requirement Categories (architectural characteristics)** — Table 7, Section 5.2.2. A two-level grouping of evaluation questions (aspect → category) spanning Boundaries, Composability, Human, Functional, Data, Timing, and Trustworthiness. Use as the standard scoring form against which every Example and use case is consistently evaluated.
- **Primitive (simple) requirements library** — Section 5.2 / 5.2.3, illustrated by the requirements-decomposition model (Figure 18). A growing set of thousands of generalized atomic requirement statements, seeded from the EPRI IntelliGrid smart-grid requirements and broadened for general CPS, each mapped to a high-level requirement category. Use to associate concrete requirements with individual use-case steps so systems can be built or tested against them.
- **Reference-architecture requirement procedure** — Section 5.2.4. After all use cases are processed, abstract each characteristic away from its source use case, group by similarity, drop duplicates, and turn the result into requirement statements levied on the Vocabulary & Reference Architecture, Timing & Synchronization, and Cybersecurity & Privacy sub-groups.

## Key Concepts
- **CPS Example vs. specific use case**: the Example is a summary with *many* systems, actors, and interactions pursuing varied goals; the specific use case has exactly *one* actor and *one* system. That cardinality difference is the defining distinction between the two stages.
- **Actor, user, system, stakeholder**: a user (actor) is any entity that interacts with the system to make it perform a function — human, machine, another system, or even the system itself responding to an internally generated trigger. A "system" here is the thing acted upon. Capturing real requirements means stepping back from actors to the wider constellation of affected parties (regulators, society, the environment) — the **stakeholders**. Table 4 lists stakeholder classes: creators, owners, operators, customers/users, supply-chain and service providers, insurers, regulators, competitors, and government.
- **Black-box vs. white-box use cases**: black-box use cases describe the actor-system interaction with no knowledge of internals; white-box use cases detail the internal workings. The templates here are black-box.
- **Architectural characteristics drive clustering**: characteristics such as data volume/velocity, variability, confidentiality, timing, and computational effort are heterogeneous, so they are grouped into the Table 7 two-level scheme. Because the count of use cases is infinite, clustering on these characteristics is what makes the analysis tractable and lets the team deliberately solicit a missing kind of use case to fill a gap.
- **Living evaluation form**: Table 7 is explicitly a *starting point*, not a complete list. When a use case shows a characteristic the form does not cover, the form is extended and the new version is applied retroactively to already-processed cases and to future ones — an iterative approach that keeps the method comprehensive.
- **Three uses of primitive requirements**: grouped by use case → the use case can be tested; grouped by system → the system can be designed and tested; grouped across everything → the reference architecture can be validated.
- **Functional requirements only**: developing use cases gathers *functional* requirements from known uses. Non-functional requirements are typically not captured (sometimes only inferred), so use-case output must be combined with other requirement-gathering methods.
- **Bidirectional sub-group coupling**: use cases tightly couple to the Vocabulary & Reference Architecture work — they both check the scope of the CPS definition and derive requirements the Framework must support, and once the definition and architecture exist, the use cases validate them. Timing, security, and data-interoperability requirements flow out to the relevant sub-groups, and any use cases those sub-groups generate flow back into the shared CPS use-case repository. These interactions were started early to avoid late gaps.

## Mental Models
- **Survey wide, then drill narrow**: Examples give breadth and tell you which concern areas are in play; specific use cases plus step-level primitives give the testable depth. One without the other is incomplete.
- **Coverage map, not a catalog**: you are not trying to list all use cases (impossible); you are trying to cover the *space* of architectural characteristics. Treat the Table 7 categories like cells of a grid and deliberately recruit use cases to fill empty cells.
- **Recognize and reuse patterns**: experienced engineers scan a collection of use cases for common aspects implementable once and reused — a PID/control loop reused across temperature and pressure deadbands, or an acceleration profile reused across an elevator, a robot arm, and a tape drive. The same pattern transfers even when the application domains differ. Interlocking collections of such patterns are what the document loosely calls the system's *architecture*: the abstract organization of a system's elements and how they connect.
- **Use cases are the goal-to-property bridge**: because each use case ties a user's goal to the system properties the CPS must exhibit (timing, security, data interoperability), they are the connective tissue between stakeholder intent and architectural requirements.
- **Build it or test it**: the same set of step-level functional requirements serves dual duty — an organization can either build a system to satisfy them or test an existing system against them.

## Worked Examples
- **Monitoring manufacturing-system energy efficiency** (Section 5.3.1). A run-to-run loop across the ISA-95-style hierarchy (enterprise planning at Level 4, manufacturing operations management at Level 3, control/automation at Level 2, sensing/actuation at Levels 1/0). Production KPIs derived from lower-level operational data yield energy-efficiency indices, which generate new configuration and scheduling parameters before the next production run. The Table 9 evaluation shows the characteristic mix: crosses many enterprise domains and resource levels; heterogeneous, often legacy subsystems; data spanning millisecond PLC I/O loops up to KPI results composed in seconds; hundreds of megabytes per run; product tolerances at parts-per-billion; example MTBF around 5,000 hours; intellectual-property loss on a confidentiality breach (recommending AES 128-bit or higher); and acting on incorrect data costing on the order of more than $1M/month. Systems/actors: the MOM application, the control-and-automation system, production equipment, and materials/personnel/energy handling units.
- **Grain/produce monitoring and delivery** (Section 5.3.2). Food producers and ingredient vendors collaborate so that lab analysis of pre-shipment samples reaches the producer in time to reject unacceptable ingredients or adjust the production process; in-transit sensors may monitor properties (temperature, humidity/moisture, light, location, biological) because ingredient quality can change during shipment. Because the trucks usually belong to a third-party logistics provider, vendor, or farmer — not the producer — sensors must repeatedly join and leave different producers' networks, over a topology that may be peer-to-peer or hub/intermediary-based. The Table 10 high-level review flags: crosses supply chain, manufacturing, transportation, and agriculture; heterogeneous subsystems including legacy labs that may only send hardcopy results; error sensitivity that can be HIGH (contamination risking illness or loss of life, so "failure is better than error"); timing from minutes of truck-monitoring resolution to minutes-to-hours of lab turnaround; synchronization needs met with timestamps; and confidentiality protecting the manufacturer's recipe with role-differentiated access for devices, farmers, lab staff, truckers, and manufacturer staff. Systems/actors: farmer, testing lab, trucker/truck, container, and customer.

## Anti-patterns
- **Inconsistent evaluation**: the source insists each Example and use case be evaluated in a consistent manner using the standard form — ad hoc, case-by-case scoring breaks the clustering and the gap analysis that the whole method depends on.
- **Treating the method as finished**: the analysis can never truly be completed; new examples and use cases keep arriving (the source expects this to *accelerate* as new CPS capabilities spark new uses). Declaring the use-case set "done" defeats its validation purpose.

## Key Takeaways
1. You cannot enumerate all CPS use cases; cover the *space of architectural characteristics* instead, clustering use cases to find and fill gaps.
2. The method is two-stage: broad **CPS Examples** (many systems/actors) for breadth, then single-actor/single-system **specific use cases** with step-level **primitive requirements** for testable depth.
3. Three templates structure the work — the CPS Example summary (Table 6), the black-box specific use case (Table 8), and the Requirement Categories evaluation form (Table 7).
4. The evaluation form is a living artifact: extend it when a new characteristic appears and re-apply it retroactively.
5. Primitive requirements are triple-purpose — test a use case, design/test a system, or validate the whole reference architecture — and use cases gather *functional* requirements only, so combine with other methods.
6. Use-case analysis is tightly and bidirectionally coupled to the Vocabulary & Reference Architecture, Timing, and Cybersecurity sub-groups, started early to prevent late gaps.
