# Chapter 2: Prepare the Organization (PO)

## Core Idea
The PO practice group ensures that an organisation's people, processes, and technology are ready to perform secure software development before and throughout the SDLC; it is the foundational enabler for every other practice group.

## Frameworks Introduced
- **PO.1 — Define Security Requirements for Software Development**: Ensure security requirements for the development infrastructure and for organisation-developed software are known, documented, and communicated to third-party suppliers at all times.
  - When to use: At programme inception and whenever requirements from internal policy, law, regulation, or major incidents change.
- **PO.2 — Implement Roles and Responsibilities**: Ensure everyone involved in the SDLC — inside and outside the organisation — has clearly defined, trained, and maintained security roles.
  - When to use: When standing up a new development team, onboarding contractors, or undergoing an annual review cycle.
- **PO.3 — Implement Supporting Toolchains**: Use automation to reduce human effort and improve the accuracy, reproducibility, and auditability of security practices; configure tools to generate artefacts evidencing those practices.
  - When to use: When selecting, deploying, or upgrading development pipeline tooling; when planning pipelines-as-code or DevSecOps automation.
- **PO.4 — Define and Use Criteria for Software Security Checks**: Define KPIs, KRIs, vulnerability severity scores, and other criteria for checking security during development; collect and safeguard the resulting evidence.
  - When to use: When establishing Definition-of-Done criteria in agile projects, or when setting up security gates in a CI/CD pipeline.
- **PO.5 — Implement and Maintain Secure Environments for Software Development**: Strongly protect all development environments (development, build, test, distribution) from internal and external threats to prevent lateral movement or compromise of software being developed.
  - When to use: When designing or auditing development infrastructure; when adopting zero-trust architecture for build systems.

## Key Concepts
- **Security Requirements (Infrastructure)**: Policies for securing the development infrastructure itself — endpoints, pipelines, environments — throughout the SDLC; reviewed at least annually or after major incidents (PO.1.1).
- **Security Requirements (Software)**: Policies specifying what the organisation's software must achieve from a security standpoint, including risk-based architecture requirements, archiving obligations, and end-of-life notification (PO.1.2).
- **Third-Party Communication (PO.1.3)**: The obligation to pass security requirements — including provenance and integrity mechanisms — through to commercial software component suppliers in contracts and acquisition documents.
- **Code Owner**: A designated individual or team responsible for each project's codebase, tasked with reviewing and approving all changes (PO.2.1 example).
- **Authorising Official Commitment (PO.2.3)**: Upper-management endorsement of secure development that is communicated to all personnel with development-related roles; a prerequisite for cultural adoption.
- **Artifact**: A piece of evidence — records generated by toolchain steps — that provide proof of secure software development practices having been performed (PO.3.3).
- **Pipelines-as-Code / Toolchains-as-Code (PO.3.2)**: Storing toolchain configuration in version-controlled code to enable reproducible, auditable builds and reduce configuration drift.
- **KPI / KRI (PO.4.1)**: Key performance indicators and key risk indicators used as measurable criteria for software security checks; should indicate how effectively security risk is being managed.
- **Environment Separation (PO.5.1)**: Network-segmented, access-controlled isolation of development, build, staging, test, and distribution environments to reduce attack surface and limit lateral movement.
- **Zero-Trust Architecture (PO.5.1/5.2)**: A hosting infrastructure model (per NIST SP 800-207) applied to development environments, requiring continuous verification rather than implicit trust based on network location.

## Mental Models
- Use PO.1 when drafting supplier contracts: translate internal security requirements into explicit contract clauses that require attestation and provenance data.
- Use PO.3 toolchain artefacts as audit evidence: if a security activity does not generate a traceable record, it cannot be demonstrated to have occurred.
- Use PO.5 environment separation to limit blast radius: an attacker who compromises one environment should not be able to traverse to another.
- Use PO.4 criteria to define "done" for security, not just functionality, so that security work is not deferred indefinitely in agile sprints.

## Anti-patterns
- **Requirements defined once and never reviewed**: PO.1 requires at least annual review, or sooner after major incidents or new regulatory requirements; stale requirements leave unaddressed risks.
- **Security roles as bolt-on job titles**: PO.2 requires roles integrated into the team with role-based training; naming someone a "security champion" without training or authority does not satisfy the practice.
- **Minimising direct human access to build systems without monitoring**: PO.5.1 requires continuous monitoring and auditing of all access attempts, not just minimisation.

## Key Takeaways
1. Security requirements for both the development infrastructure and the software itself must be explicitly defined, documented, and maintained over time.
2. Security requirements must flow to third-party suppliers through contracts, with attestation and provenance obligations.
3. Role-based security training must be provided to all personnel involved in the SDLC, with regular proficiency reviews.
4. Toolchain automation should generate artefacts that constitute auditable evidence of secure development practices.
5. Development environments must be separated, hardened, and continuously monitored; zero-trust principles apply to build infrastructure.
6. Security check criteria (KPIs, KRIs, severity scores) must be defined before development begins and tracked throughout the SDLC.

## Connects To
- **NIST SP 800-207 (Zero Trust Architecture)**: Referenced in PO.5.1 and PO.5.2 for securing development hosting infrastructure.
- **NIST SP 800-161 (C-SCRM)**: Directly supports PO.1.3 third-party security requirement communication and supply-chain risk management.
- **EO 14028 Section 4e**: PO.5.1 and PO.5.2 address EO subsections 4e(i)(A)–(F) on securing development environments.
- **NIST CSF (ID.GV, ID.SC, PR.AC, PR.AT)**: PO practices align with the CSF's Govern and Protect functions at the organisational level.
